# 두산 계열사 AI-Ready Data 현황분석 Summary

> 6개 계열사 현업 인터뷰(2026.5.21~5.29: 에너빌리티·퓨얼셀·VCC·테스나·전자BG·밥캣) 기반 진단

---

## 핵심 메시지

대부분의 계열사가 **"데이터가 어디에 있고 무엇을 의미하는지 모르는"** 단계에 머물러 있으며, 이를 책임지고 관리할 **데이터 오너십과 중앙 거버넌스가 부재**해 AI 과제가 데이터 탐색·정합성 확인에 시간 대부분을 소진하거나 PI(프로세스 혁신)로 후퇴하고 있다. 밥캣만 유일하게 데이터 오너 지정·정의 문서화 체계를 갖춰 성숙도 격차를 보인다.

---

## 1. 계열사별 진단

| 계열사 | 성숙도 | 데이터 관점 핵심 문제 | 조직·거버넌스 핵심 문제 |
|---|---|---|---|
| **에너빌리티** | 낮음 | 데이터 카탈로그 없음(위치를 감각으로 파악), 운전데이터 태그 불일치(동일 데이터가 다른 태그로 유입→수기 매핑), 전처리 기준 담당자별 상이, 옛 PDF OCR 불가, 도면 Lineage 부재(못 찾으면 처음부터 재작업) | 데이터 품질관리 프로세스 없음(외주 용역 결과 검증 불가), DataOps·변경관리 없음, 관리 주체 불명확 |
| **퓨얼셀** | 낮음 | DB화 파이프라인 서버용량 문제로 중단, 셧다운 코드 대분류만 기록→상세코드 수기 트래킹, 레이블링 규칙 없음, 버전·전처리 이력 미기록 | **IT 담당부서 자체가 없음**(담당자 퇴사 후 공백), AI 자문 조직·전사 공통 인프라 없음, 보안·클라우드 업로드 가이드 부재(물어볼 곳이 없음) |
| **VCC** | 낮음 | 시스템 파편화(Oracle·SAP·MES·RMS 분리, API 미제공), **LOT ID 연결 불가**(분할 로직 모름, 메타테이블·키 없음), 분석계 없어 쿼리 불가(뷰테이블만), 생산 capa 70~80% 수기 엑셀 | **데이터 오너십·오너 리스트 부재**(누가 무엇을 가졌는지 모름), 표준 절차·Gate 없음, PM 비전문가, DDI 비용청구 구조로 현업이 과제 회피 |
| **테스나** | 낮음~중간 | 데이터 카탈로그 없음(쿼리 매번 DDI 요청), 테이블 디스크립션 없음, 용어 불일치(제품=디바이스=자재), 70~80% 엑셀 다운받아 개인 대시보드, 정식 AI 과제 미진행(온톨로지 미흡으로 PoC 실패) | 디지털혁신팀이 오너십 보유하나 DDI 의존(추출만), AX·DE센터와 소통 부족, 온프레미스 제약 |
| **전자BG** | 중간 | DB구조·테이블 정의서 없음(DDI도 미갱신), 용어·컬럼 정의 모호(싱크에 프로젝트의 1/4~1/3 소요), MES 정합성 문제(특정 시점만 저장+수기입력 오류), 품질 데이터 대부분 엑셀(SAP보다 엑셀 신뢰) | 전체 관리 조직 없어 DDI 의존, IT 조직은 DDI에만 존재, API 미개방(보안 리스크), 담당자 잦은 교체·관리포인트 부재 *(단, 파이프라인·OCR 자체 구축으로 개선 중)* |
| **밥캣** | **높음(벤치마크)** | 10년간 Oracle DB 체계+수천 개 대시보드 운영, 재무·운영 데이터 Snowflake 통합 완료. *과제:* SAP 전환 중 미전환 데이터 수작업, 데이터 카탈로그·semantic layer 구축 중(검색 미완), 비정형·도면·PPT의 AI 정확도 및 버전관리 이슈 | **현업 데이터 오너 지정+definition 문서화·메타데이터 입력 체계 운영**, citizen development(현업이 개발자), best practice 배포. *과제:* 휴먼에러 방지 프로세스 없음, 국가별 데이터 규정 리스크 |

## 2. 공통 진단 — 데이터 관점

1. **데이터 카탈로그·메타데이터 부재** — 어떤 데이터가 어느 시스템에 있는지 알 수 없어 매번 DDI·담당자에게 의존(에너빌리티·VCC·테스나·전자BG 공통).
2. **시스템 파편화 & 연결키 부재** — Oracle·SAP·MES·RMS가 분리 운영되고 API 미제공, LOT ID 등 공통키가 없어 밸류체인 데이터 연결 불가(VCC·전자BG).
3. **표준 용어·정의 불일치** — 동일 데이터가 시스템·프로젝트·부서마다 다른 명칭/태그로 존재하고 컬럼 정의가 모호(전 계열사).
4. **품질·정합성 검증 불가** — 뷰테이블만 접근 가능해 원천 검증이 안 되고, 수기 입력 오류·외주 결과물을 거를 게이트가 없음.
5. **수기 엑셀·로컬 의존** — 중요 데이터의 70~80%가 엑셀/로컬 PC에 있고, 일부는 시스템보다 엑셀을 더 신뢰.
6. **Lineage·버전관리 부재** — 원본·최신·사용 이력 추적이 안 되어, 데이터를 못 찾으면 처음부터 재작업.
7. **전처리 표준·비정형/아날로그 데이터 방치** — 전처리가 담당자별 제각각이고, 옛 PDF·도면·사진·수기 문서가 활용되지 못한 채 축적.

## 3. 공통 진단 — 조직·거버넌스 관점

1. **데이터 오너십 부재** — 데이터를 책임지는 주체와 오너 리스트가 없어, 무엇이 문제인지·누구에게 물어야 하는지조차 모름(VCC·에너빌리티·전자BG).
2. **중앙 데이터 관리 조직 부재** — 전사를 관리하는 조직이 없어 DDI에 의존하며, 퓨얼셀·로보틱스는 IT 담당부서 자체가 없음.
3. **DDI 의존의 한계 & 비용 구조** — DDI도 테이블 내용·정의서를 보유하지 않고, 요청 시 비용이 청구되어 현업이 과제를 회피.
4. **과도한 권한·접근 통제** — 분석계 부재로 직접 쿼리 불가, API는 보안 리스크를 이유로 차단되어 PoC가 production으로 못 넘어감.
5. **표준 프로세스·Gate 부재** — feasibility 검증, 성공·평가 기준, 변경관리 프로세스가 없어 과제 추진 여부를 판단할 기준이 없음.
6. **AI·데이터 역량 및 가이드 부재** — PM이 데이터/AI 비전문가, 자문 조직이 없고, 보안·클라우드 업로드 기준이 불명확.
7. **현업 협업 단절** — 데이터 연결의 필요성을 이해하지 못하거나 엑셀 공유를 꺼려 사일로가 고착.

---

## 4. 시사점

- **공통 우선과제:** ① 데이터 카탈로그/메타데이터 + 표준 용어집 구축, ② 데이터 오너십 지정 및 중앙 거버넌스 체계 수립, ③ 과제 추진 Gate·변경관리 프로세스 정립.
- **밥캣 모델의 확산:** 현업 데이터 오너 지정 + definition 문서화 + citizen development가 성숙도 격차의 핵심 동인 → 타 계열사 적용 가능한 레퍼런스.
- **DDI 역할 재정의:** "추출 대행+비용청구" 단위에서 벗어나, 테이블 정의·메타데이터를 관리하는 데이터 관리 주체로의 전환 필요.
