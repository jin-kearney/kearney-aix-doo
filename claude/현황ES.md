# 두산 계열사 AI-Ready Data 현황 — Executive Summary

> 계열사별 사례 스토리라인(에너빌리티·VCC·테스나·전자BG·밥캣·DDI 횡단) + 퓨얼셀·CHRO 인터뷰를 묶은 한 장 요약
> 핵심: 현업은 이미 AI 과제를 시도했고, 막힌 지점은 모델·기술이 아니라 **"데이터를 찾고·꺼내고·믿는"** 데이터 기반과 그것을 책임질 **거버넌스**였다.

---

## 한눈에

여러 계열사가 자율생산·품질예측·RAG 챗봇·온톨로지 등 **AI 과제를 실제로 착수**했지만, 거의 모든 사례가 같은 자리에서 멈췄다. 전자BG는 데이터 확인에만 과제 기간의 **1/3**, 용어 싱크에 **1/4**을 썼고, VCC의 생산계획 데이터 **70~80%는 수기 엑셀**이라 LG CNS가 "현 데이터로는 AI 진행 불가" 판정을 내렸으며, 테스나의 트러블슈팅 RAG는 데이터 취합 수작업 **2주** 끝에 중단됐다. 공통 원인은 ① **데이터가 어디 있고 무엇을 뜻하는지 모르고**(카탈로그·메타데이터·용어), ② **시스템에서 꺼낼 수 없으며**(API·연결키·수기엑셀), ③ **믿을 수 있는지 검증이 안 되고**(품질·Lineage·라벨), 이를 ④ **책임질 오너십과 표준 프로세스가 없다**는 것. 그룹 IT를 일괄 운영하는 **DDI조차 "테이블에 뭐가 있는지 모른다"**는 답을 내놓는다. 반대로 **밥캣**은 현업 데이터 오너 지정·정의 문서화로 같은 문제를 상당 부분 비껴가, 거버넌스가 차이를 만든다는 점을 보여준다.

---

## 데이터 관점에서 나온 이야기

**1. "어떤 데이터가 어디 있는지 모른다" — 카탈로그·메타데이터 부재**
테스나는 쿼리를 짤 때마다 현업→디지털혁신팀→DDI 3단계를 거치고(현업 70~80%는 결국 엑셀 다운로드), 전자BG는 데이터 위치 확인에만 매번 3~4일·과제의 1/3을 쓴다. VCC는 "DDI도 어떤 쿼리에 어떤 값이 나오는지만 알지 테이블에 뭐가 있는지는 모른다", SAP 테이블 정의서는 "없다"는 답을 받았다. 밥캣마저 Snowflake semantic layer가 미완이라 "담당자만 데이터를 찾을 수 있는 단계".

**2. "같은 걸 다르게 부른다" — 표준 용어·Glossary 부재**
테스나는 제품을 디바이스·자재로 혼용하고, 에너빌리티는 발전소마다 같은 측정값에 다른 태그가 붙어 매핑 스크립트를 프로젝트마다 버린다. 전자BG는 용어·컬럼 정의가 모호해 싱크에 기간 1/4 소모.

**3. "꺼낼 방법이 없다" — API·연결키·Lineage 부재**
VCC는 MES(Oracle)↔RMS(SAP)를 잇는 LOT ID 분할 로직을 "DDI도 모른다"며 조인이 홀딩됐고, SAP API가 아예 없어 컴플라이언스 자동화도 멈췄다. 전자BG는 API 개방이 "운영 리스크"로 거부돼 파일럿→production 전환이 막혔다. 에너빌리티는 도면 버전이 어느 프로젝트에 쓰였는지 이력이 없어 "못 찾으면 처음부터 다시"한다.

**4. "중요한 건 다 엑셀·로컬에 있다" — 데이터 디지털화 부재**
VCC 라인별 capa 70~80%, 전자BG 품질 데이터 거의 전부가 수기 엑셀이고, 현업은 "SAP보다 엑셀이 더 믿을 만하다"고까지 답했다. 퓨얼셀 raw 데이터는 로컬PC, CHRO 조직도는 시스템 원본 없이 PPT로 수작업 도식화한다.

**5. "비정형은 손도 못 댄다" — 전처리·온톨로지 부재**
에너빌리티는 DWG 도면 파싱이 안 돼 온톨로지 1년 작업에 수작업이 다량, 전자BG 성적서는 PDF·이미지·엑셀·워드 4형식 혼재, 밥캣 PDF 매뉴얼 RAG 첫 구축에 6개월, 테스나는 매뉴얼을 엑셀에 캡처·붙여넣기하느라 2주가 걸렸다.

**6. "믿을 수 있는지 확인이 안 된다" — 품질·라벨·평가 부재**
VCC는 뷰테이블만 보여 "정확도 90% 넘으면 쓸만한 데이터"라는 임계치로만 운용하고 학습 데이터 오염 여부를 사후에도 모른다. 테스나 AVI는 불량 유형 라벨이 없고 진성fail 5%/가성fail 95% 불균형으로 분류 실패, 밥캣은 안전 직결 챗봇 평가셋 1,000건을 매번 수작업 구성한다. 퓨얼셀은 셧다운 상세코드가 안 나와 일지로 수기 레이블링한다.

**7. "민감정보를 어떻게 다뤄야 할지 모른다" — 비식별화·보안등급 부재** *(HR/규제 특화)*
CHRO는 비개발자라 비식별화 방법·수준을 몰라 과제마다 가명처리 항목을 매번 정의하고 복호화 경로가 없다. 밥캣은 100명이 3개국에서 에이전트를 만들며 국가별 데이터 규정 위반을 우려, 테스나는 제품군별 클라우드 반출 경계가 없어 과제마다 SA팀 승인을 반복한다.

## 조직·거버넌스 관점에서 나온 이야기

**1. "데이터를 책임지는 사람이 없다" — 오너십·오너리스트 부재**
VCC는 "데이터 오너십이 없어 뭘 찾아야 하는지조차 모르고", 데이터 오너 리스트 자체가 없다. 전자BG·테스나 AI팀도 시스템·데이터 오너십 없이 화면 단위만 본다. **대조: 밥캣은 데이터 소스별 현업 오너를 지정**해 definition 문서화·메타데이터 입력을 현업이 수행한다.

**2. "결국 다 DDI에 물어봐야 하는데, DDI도 모른다" — 중앙관리 부재 & DDI 단일 의존**
"전체를 관리하는 조직이 사내에 없어 DDI에 의존"(전자BG)하지만 DDI는 운영 주체일 뿐 데이터 내용·정의서를 갖지 않는다. 퓨얼셀·로보틱스는 IT 담당부서 자체가 없어 공식 커뮤니케이션 채널조차 없다.

**3. "요청하면 비용이 청구되니 차라리 안 한다" — DDI 과금 구조**
DDI는 KPI를 가진 하나의 유닛이라 API 개발·추출에 비용을 청구하고, 현업은 부담 때문에 과제를 포기하는 쪽으로 간다(VCC).

**4. "권한을 안 열어준다" — 분석계·API 접근 통제**
VCC는 분석계가 없어 직접 쿼리 불가(뷰테이블만), 보안상 계정도 못 연다. 전자BG는 "외부인에게 API를 여는 것 자체가 리스크"라는 이유로 막힌다.

**5. "해도 되는지 판단할 기준이 없다" — 과제 정의·Gate·표준 프로세스 부재**
DDI는 "데이터 준비를 간과한 채 결과만 요구해 과제 정의가 안 된다"고 했고, VCC는 Gate가 없어 R&D Management가 3년째 scope·비용 초과 중이다. 밥캣의 변경관리는 "방법론 수준", CHRO는 "표준 프로세스·체크리스트가 있으면 좋겠다"고 했다.

**6. "기대는 큰데 데이터 현실을 모른다" — 리터러시·임원 눈높이**
DDI는 "임원은 AI를 만능으로 보고 데이터 준비를 간과한다", VCC는 PM이 마케팅 출신 비전문가라 일정관리에 그친다고 했다. CHRO는 ERD 개념 자체가 생소해 컨설팅사에 의존한다.

**7. "지식이 사람에 묶여 있고 협업이 끊긴다" — 속인화 & 사일로**
DDI는 현업 담당자 교체주기가 시스템 담당자보다 빨라 지식이 단절된다고 했고, VCC·전자BG는 현업이 데이터 연결의 필요성을 이해 못하거나 엑셀 공유를 꺼린다. **밥캣은 현업이 곧 개발자(citizen development)**가 되어 이 간극을 메운다.

---

## 시사점

데이터 문제와 거버넌스 문제는 별개가 아니다. 카탈로그·API·디지털화가 비어 있는 근본 원인은 그것을 **누가 소유하고 어떤 기준으로 정비할지**가 정해지지 않은 데 있다. 따라서 우선순위는 ① **데이터 오너십 지정 + 카탈로그·메타데이터·표준 용어집** 구축, ② **DDI를 "추출 대행"에서 데이터 정의·위치를 관리·제공하는 주체로** 재정의, ③ **과제 Gate·변경관리·비식별화 표준 프로세스** 수립이다. 밥캣의 현업 오너십·citizen development와 CHRO의 'AI Literacy 확산' 방향이 그 레퍼런스가 된다.
