# 1. Why: 왜 필요한가  
  
## Slide 1. Why-1. AI Ready Data 관점의 필요성  
  
**핵심 메시지**  
AI 프로젝트는 원천 데이터만으로 시작되지 않으며, 모델이 학습하고 Agent가 판단하려면 사람의 업무 판단 기준이 데이터 위에 구조화되어 있어야 한다.  
  
**장표 본문**  
  
| AI 활용 영역 | 필요한 해설/주석 | 기대 효과 |  
|---|---|---|  
| 분류 모델 | 문의 유형, 불만 유형, 정상/이상, 리스크 등급 | 학습 가능한 정답 라벨 확보 |  
| 예측 모델 | 타깃 이벤트 발생 여부, 기준 시점, 양성/음성 판정 | 타깃 정의 오류 방지 |  
| RAG | 문서 유형, 주제, 조항, 근거성, 민감도 | 검색 정확도 및 답변 근거성 향상 |  
| Agent | 업무 의도, 상태, 다음 액션, 예외 상황, 승인 필요 여부 | 자동 분기와 업무 판단 정확도 향상 |  
| LLM 평가 | 사실성, 근거성, 유해성, 선호 응답, 수정 답변 | 평가셋 및 튜닝 데이터 확보 |  
  
**강조 문구**  
  
> 데이터 해설/주석은 사람이 암묵적으로 판단하던 업무 기준을 AI가 활용 가능한 데이터 자산으로 전환하는 작업이다.  
  
**핵심 흐름**  
  
Raw Data → Annotated Data → Training/Evaluation Dataset → AI Model/RAG/Agent  
  
---  
  
## Slide 2. Why-2. 해설/주석 부재 시 발생하는 문제  
  
**핵심 메시지**  
해설/주석 체계가 없으면 AI 프로젝트는 라벨 기준을 매번 새로 만들고, 주석 품질이 흔들리며, 모델 오류의 원인을 추적하기 어려워진다.  
  
**장표 본문**  
  
| 문제 | 구체적 증상 | AI/Data 프로젝트 영향 |  
|---|---|---|  
| 정답 데이터 부재 | 학습·평가 기준이 되는 라벨 없음 | 모델 학습 지연, PoC 반복 |  
| 라벨 기준 불명확 | 주석자마다 다른 기준으로 태깅 | 학습 데이터 불일치, 모델 성능 불안정 |  
| 업무 판단 암묵지 의존 | 전문가의 머릿속 기준만 존재 | 담당자 변경 시 품질 저하 |  
| 품질 검수 미흡 | 오분류, 누락, 과잉 라벨링 미탐지 | 잘못된 Ground Truth로 모델 학습 |  
| 편향된 데이터셋 | 특정 클래스, 상황, 고객군 과대표집 | 모델 편향 및 일반화 성능 저하 |  
| 외주 라벨링 통제 부족 | 작업 기준, 검수 기준, 재작업 기준 모호 | 비용 증가 및 품질 책임 불명확 |  
  
**강조 문구**  
  
> 라벨 품질은 모델 품질의 상한선을 결정한다.  
  
---  
  
## Slide 3. Why-3. AI 프로젝트 사전 준비 효과  
  
**핵심 메시지**  
계열사 공통 해설/주석 체계는 AI 프로젝트 착수 전 정답 기준, 평가 기준, 업무 판단 기준을 미리 준비해 프로젝트 속도와 재사용성을 높인다.  
  
**장표 본문**  
  
| 사전 준비 항목 | AI 프로젝트 효과 | 운영 효과 |  
|---|---|---|  
| 공통 라벨 체계 | 과제별 라벨 정의 반복 감소 | 계열사 간 비교 가능성 확보 |  
| Annotation Guideline | 주석자 판단 차이 감소 | 외주·내부 작업 기준 통일 |  
| Gold Set | 모델 평가 기준 확보 | 품질 검수 기준 명확화 |  
| Annotated Dataset | 학습/검증/평가 데이터 즉시 활용 | 데이터셋 재사용 가능 |  
| Human Feedback Dataset | LLM 평가·튜닝 데이터 확보 | 지속 개선 루프 구축 |  
| 검수·승인 이력 | 데이터셋 신뢰도 확보 | 감사 및 책임 추적 가능 |

# 2. What: 무엇인가  
  
## Slide 4. What-1. B-2 데이터 해설/주석의 정의  
  
**핵심 메시지**  
B-2 데이터 해설/주석은 컬럼 설명이 아니라, AI가 학습·평가·검색·판단에 활용할 수 있도록 데이터에 라벨, 태그, 판정값, 근거, 맥락 정보를 부여하는 체계이다.  
  
**장표 본문**  
  
| 구분 | 내용 |  
|---|---|  
| 잘못된 해석 | Field Description, 컬럼 설명, 데이터 사전 항목 |  
| 올바른 해석 | Data Annotation, Data Labeling, Human Feedback Dataset 관리 |  
| 핵심 목적 | 원천 데이터에 업무적 해석과 정답 기준을 부여하여 AI 활용 가능한 데이터셋으로 전환 |  
| 주요 산출물 | Label Taxonomy, Annotation Guideline, Annotated Dataset, Gold Set, 검수 이력 |  
| 적용 시점 | AI 프로젝트 착수 전, 학습/평가/RAG/Agent 데이터셋 준비 단계 |  
  
**강조 문구**  
  
> 데이터 해설/주석은 “데이터 구조 설명”이 아니라 “AI가 사용할 판단 기준을 데이터화하는 작업”이다.  
  
---  
  
## Slide 5. What-2. 관리 대상과 적용 예시  
  
**핵심 메시지**  
데이터 해설/주석은 텍스트뿐 아니라 문서, 이미지, 영상, 음성, 센서, 거래, 로그, LLM 응답까지 확장 적용된다.  
  
**장표 본문**  
  
| 데이터 유형 | 해설/주석 예시 | 대표 AI 활용 |  
|---|---|---|  
| 고객 상담 텍스트 | 문의 유형, 감정, 해지 의도, 해결 여부 | 상담 자동 분류, 상담 Agent |  
| VOC/리뷰 | 주제, 긍·부정, 긴급도, 원인 카테고리 | VOC 분석, 제품 개선 |  
| 계약/문서 | 조항 유형, 리스크 조항, 의무사항, 근거 문장 | 문서 AI, 계약 리스크 분석, RAG |  
| 이미지 | 객체 위치, 결함 유형, 심각도, 정상/불량 판정 | 품질 검사, 안전 모니터링 |  
| 영상 | 이벤트 구간, 객체 추적, 위험 행동 | CCTV 분석, 제조 안전 |  
| 음성/콜 | 발화자, 의도, 감정, 컴플라이언스 위반 | 상담 품질 평가, STT 후처리 |  
| 센서/시계열 | 정상/이상 구간, 고장 전조, 원인 이벤트 | 이상탐지, 예지보전 |  
| 거래/로그 | 이상거래 여부, 사기 의심 사유, 처리 상태 | Fraud Detection, 운영 자동화 |  
| LLM 응답 | 선호 응답, 오류 유형, 사실성, 근거성 | LLM 평가, Fine-tuning, RLHF/DPO |

**핵심 메시지**  
B-2는 데이터 구조 설명이 아니라, 데이터 포인트에 대한 업무적 판정과 AI 학습·평가용 해석 결과를 관리하는 영역이다.  
  
**장표 본문**  
  
| 포함 범위 | 설명 |

| 주석 가이드라인 작성 | 주석 기준, 예시, 예외, 애매한 사례 처리 기준 정의 |  
| 주석 작업 운영 | 대상 선정, 작업 배분, 일정, 생산성 관리 |

| Annotated Dataset 관리 | 학습/검증/평가 데이터셋 버전 및 이력 관리 |  
| Human Feedback 수집 | LLM 응답 평가, 선호도 비교, 수정 응답, 사용성 피드백 |  
  
| 제외 범위 | 설명 |  
|---|---|

| 데이터 카탈로그 | 데이터 위치, 오너, 접근 경로 관리는 별도 주제 |  
| 일반 데이터 품질 규칙 | 결측률, 중복률, 정합성 검증은 데이터 품질 관리 영역 |  
| 모델 학습 알고리즘 | 모델 구조, 하이퍼파라미터, 학습 파이프라인은 MLOps 영역 |

| 데이터 카탈로그 | 데이터 위치, 오너, 접근 경로 | B-2는 AI 학습용 해석 데이터셋 관리 |

| MLOps | 모델 학습, 배포, 모니터링 | B-2는 학습·평가 데이터셋 품질 운영 |

# 4. How: 어떻게 구축하는가  
  
## Slide 10. How-1. 표준 구축 프로세스  
  
**핵심 메시지**  
데이터 해설/주석은 라벨 체계 설계, Pilot Annotation, 품질 진단, 본 주석, 검수, 재사용 등록의 단계로 운영되어야 한다.  
  
**장표 본문**  
  
| 단계 | 주요 활동 | 산출물 |  
|---|---|---|  
| 1. 과제 정의 | AI 활용 목적과 주석 필요성 확인 | Annotation Use Case 정의서 |  
| 2. 데이터 샘플링 | 대표성 있는 데이터 추출, 민감정보 점검 | Pilot Sample Set |  
| 3. 라벨 체계 설계 | 클래스, 태그, 속성, 판정 기준 정의 | Label Taxonomy |  
| 4. 가이드라인 작성 | 예시, 예외, 애매한 사례 기준 작성 | Annotation Guideline |  
| 5. Pilot Annotation | 소량 샘플 주석 및 기준 테스트 | Pilot Annotated Dataset |  
| 6. 품질 진단 | 일치율, 오류 유형, 기준 모호성 분석 | Pilot QA Report |  
| 7. 기준 보완 | 라벨 통합/분리, 예시 추가, 제외 기준 보완 | Guideline v1.0 |  
| 8. 본 주석 수행 | 작업 배분, 생산성 관리, 이슈 처리 | Annotated Dataset |  
| 9. 검수·재작업 | 샘플/전수 검수, Consensus, Adjudication | Validated Dataset |  
| 10. 등록·재사용 | 학습/평가/RAG/Agent용 데이터셋 등록 | Training/Eval Dataset, Data Card |

## Slide 11. How-2. 라벨 체계 설계 기준  
  
**핵심 메시지**  
좋은 라벨 체계는 클래스 이름뿐 아니라 제외 기준, 우선순위, 애매한 사례, 근거 기록 기준까지 포함해야 한다.  
  
**장표 본문**  
  
| 설계 요소 | 설명 | 예시 |  
|---|---|---|  
| 클래스 정의 | 라벨별 의미와 포함 조건 | 불만, 문의, 요청, 칭찬 |  
| 제외 기준 | 라벨을 부여하지 않는 조건 | 정보 부족, 중복, 테스트 데이터 |  
| 계층 구조 | 대분류-중분류-소분류 구조 | 품질 문제 > 배송 파손 > 포장 불량 |  
| 다중 라벨 기준 | 여러 라벨 동시 부여 가능 여부 | 불만 + 해지 의도 |  
| 우선순위 규칙 | 충돌 시 어떤 라벨을 우선할지 | 안전 이슈가 일반 불만보다 우선 |  
| 애매한 사례 처리 | 판단이 어려운 경우의 기준 | 보류, 전문가 검토, 기타 금지 여부 |  
| 근거 기록 | 라벨 판정의 근거 문장·구간·영역 | 상담 발화, 문서 조항, 이미지 영역 |  
  
**강조 문구**  
  
> 라벨명만 정의하면 부족하다. 실제 품질은 예외와 경계 사례를 얼마나 명확히 정의했는지에 좌우된다.

## Slide 12. How-3. Annotation Guideline 구성  
  
**핵심 메시지**  
Annotation Guideline은 주석자 간 판단 차이를 줄이기 위한 운영 기준서이며, 예시와 예외 사례가 핵심이다.  
  
**장표 본문**  
  
| 구성 항목 | 포함 내용 |  
|---|---|  
| 과제 목적 | 왜 이 주석을 하는지, 모델/업무에서 어떻게 쓰이는지 |  
| 라벨 정의 | 각 라벨의 의미, 포함/제외 기준 |  
| 대표 예시 | Positive, Negative, Boundary Case |  
| 작업 절차 | 데이터 확인, 라벨 선택, 근거 입력, 제출 방식 |  
| 품질 기준 | 정확도, 일관성, 근거 기재, 누락 방지 |  
| 금지 사항 | 추측 금지, 개인정보 임의 복사 금지, 임의 라벨 생성 금지 |  
| 이슈 처리 | 애매한 사례, 시스템 오류, 신규 라벨 후보 보고 방식 |  
| 변경 이력 | 라벨 정의와 예시가 언제 어떻게 바뀌었는지 |  
  
**예시 템플릿**  
  
| 라벨 | 정의 | 포함 사례 | 제외 사례 | 애매한 경우 처리 |  
|---|---|---|---|---|  
| 해지 의도 | 고객이 서비스 종료 또는 계약 해지를 명시·암시 | “해지하고 싶어요” | 단순 불만 | 해지 가능성 언급 시 전문가 검토 |

## Slide 13. How-4. 주석 품질관리 체계  
  
**핵심 메시지**  
주석 품질은 단순 검수가 아니라 Gold Set, Consensus, 일치율, Rule-based Validation, Dataset Health Check를 조합해 관리해야 한다.  
  
**장표 본문**  
  
| 품질관리 방식 | 목적 | 적용 상황 |  
|---|---|---|  
| Gold Set 검증 | 전문가 기준 데이터 대비 정확도 측정 | 외주/대량 작업, 반복 품질 측정 |  
| Consensus | 동일 데이터를 여러 주석자가 라벨링하고 합의 | 주관성이 있는 텍스트/이미지 판단 |  
| Inter-Annotator Agreement | 주석자 간 일치율 측정 | 라벨 기준 명확성 진단 |  
| Adjudication | 불일치 사례를 상위 전문가가 최종 판정 | 고위험·애매한 사례 |  
| Rule-based Validation | 라벨 누락, 잘못된 값, 구조 오류 자동 탐지 | 형식 오류와 논리 오류 탐지 |  
| Dataset Health Check | 클래스 분포, 대표성, 편향, 희소 클래스 확인 | 학습 데이터셋 품질 점검 |  
| Model Error Review | 모델 오류 샘플을 재검토 | 모델 개선 루프 |  
  
**품질관리 계층**  
  
1. Label-level QA: 개별 주석 정확성 검수  
2. Annotator-level QA: 주석자 간 일치율 및 작업자 품질 측정  
3. Dataset-level QA: 클래스 분포, 대표성, 편향 점검  
4. Workflow QA: 기준 변경, 재작업, 승인, 감사 이력 관리

## Slide 14. How-5. 운영 및 변경관리 체계  
  
**핵심 메시지**  
주석 데이터셋은 일회성 산출물이 아니라 라벨 체계, 가이드라인, 데이터셋, 모델 성능 피드백이 함께 버전 관리되는 운영 자산이다.  
  
**장표 본문**  
  
| 운영 프로세스 | 설명 | 관리 포인트 |  
|---|---|---|  
| 신규 주석 요청 | AI 과제 또는 데이터셋 후보 발생 시 필요성 평가 | 주석 필요성, 예상 규모, 활용 목적 |  
| 라벨 체계 승인 | 업무 오너가 신규 라벨과 기준 승인 | 기준 책임자 명확화 |  
| 작업 배정 | 내부 전문가, 스튜어드, 외부 라벨러에게 작업 배분 | 권한, 일정, 생산성 |  
| 주석 수행 | 라벨, 태그, 근거, 점수, 피드백 입력 | 가이드라인 준수 |  
| 품질 검수 | Gold Set, Consensus, 샘플 검수, 자동 검증 | 오류 유형 기록 |  
| 재작업 및 교육 | 오류 패턴 기반 재작업과 주석자 보정 | 재발 방지 |  
| 버전 등록 | 학습/평가/운영용 데이터셋 버전 등록 | Dataset Version 관리 |  
| 활용 피드백 | 모델 오류와 Agent 실패 사례를 기준 개선에 반영 | 지속 개선 루프 |  
  
**버전 관리 대상**  
  
- Label Taxonomy Version  
- Annotation Guideline Version  
- Annotated Dataset Version  
- Gold Set Version  
- Model/Evaluation Version

## Slide 15. How-6. 자동화 가능 영역  
  
**핵심 메시지**  
LLM과 모델 자동화는 주석을 대체하는 것이 아니라, 초안 생성·샘플 추천·품질검증을 보조하여 Human-in-the-loop 효율을 높이는 역할을 한다.  
  
**장표 본문**  
  
| 자동화 방식 | 기능 | 운영상 의미 |  
|---|---|---|  
| Pre-labeling | 기존 모델이나 LLM이 초안 라벨 생성 | 사람이 수정하는 검수 중심 운영으로 전환 |  
| Active Learning | 모델이 불확실하거나 정보량이 큰 샘플 추천 | 적은 라벨로 성능 개선 |  
| Weak Supervision | 규칙, 키워드, 외부 지식, 기존 모델로 약한 라벨 생성 | 대량 데이터에 초기 라벨 생성 |  
| LLM-assisted Annotation | LLM이 라벨, 근거 문장, 요약 주석 제안 | 도메인 전문가 생산성 향상 |  
| Auto QA | 누락, 형식 오류, 라벨-근거 불일치 탐지 | 품질 검수 효율화 |  
| Human Feedback Loop | 모델 오류 샘플을 다시 주석·검수해 학습 반영 | 지속 개선 루프 구축 |  
| LLM-as-a-Judge 보조 | LLM이 응답 품질을 1차 평가하고 사람이 샘플 검수 | LLM 평가 데이터 확장 |  
  
**운영 원칙**  
  
> AI 생성 라벨은 초안이다. 공식 데이터셋은 사람 검수와 승인 후 확정한다.
