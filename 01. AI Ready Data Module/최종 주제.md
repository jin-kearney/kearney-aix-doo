A-1	데이터 카탈로그	AI와 사용자가 데이터 자산의 존재, 위치, 오너, 접근 경로를 찾을 수 있도록 등록하는 자산 목록 체계	"어디에 무엇이 있는가"만 다룸. 의미는 A-2에서	1. 어떤 데이터 자산을 카탈로그에 등록해야 하는가?	데이터 자산 등록 대상 기준, 우선 등록 데이터 목록	모든 데이터를 한 번에 등록하지 않고, AI 활용 가능성이 높거나 반복 사용되는 핵심 데이터부터 등록한다.	계열사별 주요 데이터 자산을 조사하고, 시스템 데이터, 문서, 보고서, 품질 데이터, 실험 데이터, 설비 데이터 등 등록 대상 범위를 정의한다. 모든 데이터를 한 번에 등록하지 않고 AI 활용 가능성, 업무 중요도, 재사용 가능성 기준으로 우선순위를 정한다.

				4. AI나 사용자가 필요한 데이터를 어떻게 쉽게 찾게 할 것인가?	검색 분류 체계, 데이터 탐색 화면 구성 기준	업무 도메인, 데이터 유형, 보유 조직, 시스템 기준으로 탐색 가능하게 구성한다.	업무 도메인, 데이터 유형, 보유 조직, 시스템, 활용 목적 기준으로 검색 구조를 설계한다. 단, 여기서는 데이터의 의미 해석이 아니라 데이터 자산을 찾기 위한 분류와 탐색 구조만 다룬다.
				5. 카탈로그를 최신 상태로 어떻게 유지할 것인가?	카탈로그 갱신 프로세스, 미등록 데이터 점검 기준	신규 생성, 폐기, 위치 변경, 오너 변경이 발생하면 카탈로그 갱신이 필요하다는 원칙을 둔다.	신규 데이터 생성, 데이터 폐기, 저장 위치 변경, 오너 변경이 발생했을 때 카탈로그가 갱신되는 절차를 정의한다. 자동 감지, 오너 확인, 정기 점검 방식을 포함한다.

AI-ready 메타데이터 표준	데이터 타입, 단위, 생성 시스템, 갱신 주기, 오너, 사용 조건 등 AI 해석에 필요한 속성을 정의한다.	테이블명, 컬럼명, 데이터 타입, 단위, 포맷, 허용값, 생성 시스템, 갱신 주기, 데이터 오너 등 AI가 데이터를 해석하기 위해 필요한 기술·운영 속성을 정의한다.

				3. 테이블명과 필드명만으로 의미를 알 수 없는 데이터를 어떻게 설명할 것인가?	필드 설명서, 컬럼 Dictionary	약어와 코드 중심 필드는 자연어 설명을 붙여 AI와 사용자가 해석 가능하게 한다.	약어, 코드명, 현장식 필드명에 대해 자연어 설명을 붙인다. 예를 들어 DEF_CD는 단순 코드명이 아니라 “품질 결함 유형을 식별하는 코드”처럼 필드 수준 설명을 작성한다.

A-3	비즈니스 Glossary	조직 내 업무 용어, 약어, 동의어를 표준 정의로 통일해 데이터 해석의 일관성을 확보하는 용어 사전	단어 단위 정의만. 개념 간 관계는 B-3	1. 어떤 용어를 표준화 대상으로 삼을 것인가?	표준화 대상 용어 목록	AI 검색과 해석에 영향을 주는 결함명, 검사항목, 공정명, 원인 유형, 조치 유형 등을 우선 표준화한다.	부서·계열사·문서별로 다르게 쓰이는 핵심 업무 용어를 식별한다. 결함명, 검사항목, 공정명, 제품군, 원인 유형, 조치 유형, 고객 불만 유형처럼 AI 검색과 해석에 영향을 주는 용어를 우선 대상으로 한다.
				2. 표준 용어는 어떤 항목으로 정의해야 하는가?	Glossary 표준 템플릿	표준명, 영문명, 약어, 동의어, 금지어, 정의, 사용 예시를 최소 항목으로 둔다.	표준 용어, 영문명, 약어, 동의어, 금지어, 정의, 사용 예시, 관련 데이터 필드, 책임 부서를 정의한다. AI가 현장 용어를 표준 용어로 변환할 수 있도록 동의어와 약어를 함께 관리한다.
				3. 계열사나 부서 간 용어 충돌을 어떻게 조정할 것인가?	용어 충돌 조정 프로세스	동일 개념을 다르게 부르거나 동일 용어를 다르게 쓰는 경우 승인·조정 절차를 둔다.	같은 단어가 다른 의미로 쓰이거나, 다른 단어가 같은 의미로 쓰이는 사례를 조정하는 절차를 정의한다. 현업 SME, 데이터 오너, 중앙 데이터 조직이 참여하는 승인 구조를 둔다.
				4. Glossary를 AI 활용 체계와 어떻게 연결할 것인가?	Glossary-AI 검색 연계 방식	사용자가 현장 용어로 질문해도 AI가 표준 용어로 변환해 관련 데이터를 찾도록 연결한다.	Glossary를 카탈로그 검색, 메타데이터, RAG 검색, Prompt, Agent와 연결한다. 예를 들어 사용자가 현장식 결함명을 입력하면 AI가 표준 결함명으로 변환해 관련 데이터를 찾게 한다.
				5. 용어 변경을 어떻게 관리할 것인가?	용어 변경 관리 기준	신규 용어 등록, 기존 용어 변경, 폐기 용어 지정 시 승인과 버전 관리를 적용해 AI 검색과 데이터 해석의 일관성을 유지한다.	신규 용어 등록, 기존 용어 변경, 폐기 용어 지정, 동의어 추가에 대한 버전 관리와 승인 절차를 정의한다. 용어 변경이 관련 데이터, 문서, Prompt에 미치는 영향도 함께 점검한다.
B-1	데이터 전처리	문서, 표, 이미지 등 비정형·반정형 데이터를 AI가 읽을 수 있는 구조화 형태로 변환하는 작업	형식 변환만. 품질 검증은 C-2, OCR 등 대량전환(디지털화)는 F-3	1. 어떤 데이터가 전처리 대상인가?	전처리 대상 데이터 유형 목록	PDF, PPT, Excel, 이미지, 표, 반정형 보고서처럼 AI가 바로 읽기 어려운 디지털 자료를 대상으로 한다.	PDF, PPT, Excel, 이미지, 표, 스캔된 디지털 파일, 반정형 보고서처럼 AI가 바로 읽기 어려운 데이터를 식별한다. 이미 디지털 형태로 존재하지만 구조화가 안 된 자료를 대상으로 한다.
				2. 문서 유형별로 무엇을 추출해야 하는가?	문서 유형별 추출 기준	문단, 표, 이미지 내 텍스트, 수치, 섹션, 셀 구조 등 AI 활용에 필요한 구조 요소를 추출한다.	PDF에서는 문단, 표, 이미지, 페이지 구조를 추출하고, PPT에서는 슬라이드 제목, 본문, 도형 텍스트, 표를 추출한다. Excel에서는 시트, 셀, 수식, 병합 셀 구조를 해석한다.
				3. 추출된 내용을 어떤 구조로 변환할 것인가?	구조화 데이터 포맷, Chunking 기준	텍스트, 표, 수치, 원본 위치 정보를 보존해 검색·분석 가능한 구조로 변환한다.	텍스트, 표, 수치, 이미지 설명, 문서 섹션, 항목명을 JSON, 테이블, 청크, 인덱스 등 AI가 활용 가능한 구조로 변환한다. 원본 위치 정보도 함께 남겨 재참조가 가능하게 한다.
				4. 원천 문서 양식이 바뀌어도 전처리 결과를 어떻게 안정적으로 유지할 것인가?	전처리 테스트셋, 파서 변경 관리 기준	양식 변경 시 전처리 로직 검토가 필요하다는 원칙을 둔다.	보고서 양식 변경, 셀 구조 변경, 표 레이아웃 변경이 발생했을 때 전처리 로직이 깨지지 않도록 파서 버전, 테스트 문서셋, 예외 처리 기준을 둔다.
				5. 전처리 결과를 AI 활용 환경에 어떻게 제공할 것인가?	AI 입력용 전처리 산출물	전처리 결과는 검색 인덱스, Vector DB, 데이터레이크, 분석 테이블 등 AI 활용 환경에 맞는 형태로 제공한다.	전처리 결과를 검색 인덱스, Vector DB, 데이터레이크, 분석 테이블 등에 적재하는 방식을 정의한다. 단, 적재 후 품질 통과 여부는 C2에서 판단한다.
B-2	데이터 해설/주석	AI 학습을 위해 데이터에 라벨, 분류, 해석 기준 등 사람이 부여한 의미 정보를 붙이는 체계	학습 데이터 라벨만. 평가용 정답은 E-3	1. 어떤 데이터에 라벨이나 주석이 필요한가?	주석 대상 데이터 목록	이미지, 클레임 문장, 원인 분석 문서, 실험 결과처럼 사람의 해석이 필요한 데이터를 대상으로 한다.	결함 이미지, 고객 클레임 문장, C/S Report 원인 분석, 실험 결과, 조치 이력처럼 AI가 학습하려면 사람의 해석이 필요한 데이터를 식별한다.
				2. 라벨 체계는 어떻게 설계할 것인가?	라벨 Taxonomy, 라벨 정의서	결함 유형, 원인 유형, 조치 유형 등 AI가 학습할 분류 체계를 업무 기준에 맞게 정의한다.	결함 유형, 원인 유형, 조치 유형, 판정 결과 등 AI가 학습할 분류 체계를 정의한다. 라벨 간 경계, 상호 배타성, 다중 라벨 허용 여부를 명확히 한다.
