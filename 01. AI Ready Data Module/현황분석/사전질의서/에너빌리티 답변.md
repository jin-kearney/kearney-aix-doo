질문	답변
AI/Data 과제를 수행하면서 가장 어려웠던 점은 무엇이었나요?	"1. 조직 내 참여도 낮음                                                                                                                                                                                                                                                         일부 인원을 제외하면 팀 전체의 실질 참여도가 낮은 반면, 관심도 자체는 높은 편입니다. 주로 AI/Data 기초 지식 부족, 새로운 기술에 대한 막연한 불안감, 기존 도메인 지식과 AI 방법론 간 연결 고리 부재로 인해 참여도가 낮은 것 같습니다. 접근 방법을 모르고, 사내에 자문할 전문가를 찾기 쉽지 않고요
                                                                                                                                                                                                                                                                    2. 사내 AI 활용 환경의 제약                                                                                                                                                                                                                                                                    현재 사내 AI 서비스가 차단되고 MS Copilot / 두산 AI Chat(sLLM)만 허용된 환경에서는 부여받은 AI 과제를 실질적으로 수행하기 어렵습니다. 결과적으로 각 조직이 개별적으로 외주 업체를 수배하고 계약하여 개발을 추진하는 구조가 고착화되고 있습니다. 유사한 과제가 많으니 투자비용은 중복되고, 외주개발로 인한 자체 AI 역량 축적의 기회를 잃어버리고 있다고 봅니다.

3. 데이터 전처리, 라벨링 단계의 도구 부재
자체 수행 시 품질 불량 데이터의 유형 분류와 엔티티 추출 과정에서 활용 가능한 표준 툴(라벨링 자동화 도구)이 없어 수작업에 의존하게 됩니다. 도메인 전문가가 별도로 투입되지 않거나, 일관성 문제로 라벨링 품질이 낮아지고, 결과 성능 부실로 직결되는 악순환이 발생합니다. "
	"4. Function 간 데이터 카탈로그 및 메타데이터 표준 부재
Function별로 온프레미스 시스템이 독립 운영되어 조직 간 데이터 연계가 사실상 불가능한 구조입니다. 개별 과제는 추진되더라도 전사 차원의 시너지나 교차 분석은 시도조차 못하고 있습니다. 예를 들면 건설현장(공종 단위 관리)와 제작공장(PO단위 관리)간의 데이터 연결고리가 없어, 라벨링에 애로가 있었습니다. 또 다른 예로 설계도서와 연결고리가 없는 문제도 있습니다.

5. AI 개발시 외주 의존 구조의 비효율
업체마다 자체 솔루션을 제안하므로 비용과 기간은 누적되는 반면, 실무 활용형 산출물보다 시연 및 보고용 결과물에 그치는 경우가 많습니다. 유지보수 단계에서도 재차 예산이 투입되는 구조가 반복되고 있습니다.

6. 제한적인 개발 참여(Human in the loop)
외주 개발 시 실무 인력의 참여가 제한적이며, ""내 업무가 아니다""라는 인식 하에 데이터만 전달되는 경우가 많습니다. 그 결과 도메인 지식이 데이터셋과 산출물에 충분히 반영되지 못하고 있고, 최적의 데이터 구조와 접근 방식을 탐색할 수 있는 환경 자체가 조성되지 못하고 있습니다."
AI/Data 과제에 필요한 데이터를 정의하거나 식별하는 과정은 원활했나요?	1. 초기 데이터 정의한 것에 비해 개발과정에서 데이터 정의가 바뀌는 경우가 계속 발견됩니다. 특히 도메인 지식 담당자가 오기 전, 오고 나서 변경되는 부분이 많습니다. (담당자별 해석의 차이도 다수 존재)
AI/Data 과제 수행에 필요한 데이터는 어떤 방식으로 확보하셨나요? 이 과정에서 주요 병목은 무엇이었나요?	1. ERP, 수기 관리 엑셀, 사내 시스템, Sharepoint/Teams에서 취합하였습니다. 사내 시스템에는 Api가 없고, sql또한 보안등으로 제약되어 매번 다운(CSV extract)하거나 필요시 RPA를 직접 만들어 사용하였음, 그나마 MS 기반 데이터 는 api가 있어 원활하게 사용하였습니다.
필요한 데이터를 찾고, 이해하고, 접근하기 위한 관리 체계가 충분한지 의견을 작성해 주세요.	1. 타 조직이 보유한 데이터의 위치와 관리 주체를 확인하기 위해 별도의 사전 조사가 매번 선행되어야 합니다. 전사 차원의 데이터 카탈로그나 검색 기반이 없어, 필요한 데이터의 존재 여부조차 인지하기 어려운 경우가 많습니다. 결과적으로 데이터 요청 시마다 담당자를 수소문해야 하며, 시스템에 등록된 데이터보다 부서별로 수기 관리되는 파일(엑셀 등)이 더 정확하고 최신인 사례가 다수 존재합니다.
"AI/Data 과제에서 데이터의 신뢰도나 활용 가능성을 낮추는 품질 이슈는 무엇이었나요?
(AI과제에서 “이 데이터는 믿고 쓰기 어렵다”고 판단된 경우 등)"	"1. 입력 단계에서 충실하지 못하게 등록된 케이스가 다수 존재합니다. 대표적으로 본문에 ""첨부 참조""만 기재되고 실제 텍스트 데이터는 부재한 경우로, 비정형 데이터 기반 레코드는 사실상 결측치로 처리되어 유효 표본이 크게 축소됩니다. 

2. 판정의 기준값, 허용 범위, 적용 규격이 데이터에 함께 저장되어 있지 않아 ""부적합""으로 라벨링되어 있으나, 동일한 수치라도 적용 설계도서, 국제 규격(ASME, IEC 등), 내부 사양에 따라 적합/부적합이 달라지므로, 외부 지식 기반(설계도서, 규격, 기술 논문 등)이 함께 연계되지 않으면 모델이 판정 로직 자체를 학습할 수 없는 구조입니다. 결과적으로 라벨은 존재하나 라벨링의 근거가 누락된 상태로, 모델 신뢰성과 설명 가능성 모두 확보가 어렵습니다."
보유한 데이터를 AI 과제에 적용하기 위해 추가로 보완이 필요한 부분이 무엇이었나요?	1. 사전에 표준화된 도메인의 온톨로지나 데이터 모델이 없어, 매 과제마다 엔티티 정의가 필요하며 분석 목적에 맞는 엔티티와 속성, 관계를 정의하기 위해 도메인 전문가와의 반복적 협의 과정이 과제 전체 일정의 상당 부분을 소요하고 있습니다.
수집된 데이터를 AI 학습데이터로 바로 활용하기 어려웠다면 그 이유와 사례를 작성해주세요	"1. 비정형 데이터를 Rule-Base 분류시 한계 존재. 작성자별로 상이한 표현으로 기록되어 있어, 정규화 과정에서 상당한 공수가 발생 하였습니다.

2. 다수의 핵심 정보가 본문이 아닌 첨부 파일(PDF, 도면, 사진, 엑셀 등)에 포함되어 있으나, 이를 자동으로 파싱, OCR 연계할 수 있는 표준 처리 체계가 없어 학습 데이터에서 사실상 배제되고 있습니다. 특히 도면, 검사 성적서 이미지 등 시각 정보는 도메인 판단의 근거가 되는 핵심 자산임에도 불구하고 디지털 자산화가 미흡합니다.

3. 제품 PO, 제품명, 호기, KKS 등 시스템 간 연결의 기준이 되는 주요 식별자가 데이터에 누락되어 있거나, 시스템별로 상이한 체계를 사용하고 있어 데이터 간 관계 형성이 불가능한 사례가 다수 발생합니다. 결과적으로 단일 시스템 내 분석은 가능하지만, 교차 분석을 통한 인사이트 도출은 차단되는 구조입니다."
확보한 데이터를 AI 학습에 쓸 수 있는 형태로 만들기까지 어떤 작업이 필요했나요? 그 중 가장 많은 시간이 걸린 것은 무엇인가요?	1. Excel, Word, PDF, HWP 등 파일의 파서, OCR 작업이 적절한 품질의 데이터를 얻는데 까지 가장 오래 걸렸으며, 비정형 데이터는 전처리, 라벨링, 엔티티 추출, 메타데이터 생성 과정에 수작업 진행하기보다 LLM 활용하여 트라이 앤 에러 방식으로 선 진행 후 검토를 진행 하였습니다. (제가 진행한 Case에서 Fine Tune 모델 개발은 불필요)
운영 중인 과제가 있다면, 데이터 품질 저하나 모델 성능 변화를 어떻게 감지하고 대응하고 있나요?	1. 별도 모니터링 체계 없으며 현업 사용자가 직접 이상여부 알려주는 방식입니다.
AI 학습에 사용한 데이터의 출처·버전·전처리 이력을 추적하고 관리하고 있나요?	1. 별도 관리하고 있지 않습니다.
향후 AI/Data 활용을 확대하기 위해 그룹 차원에서 우선적으로 지원이 필요한 영역에 대해 자유롭게 작성해 주세요	"1. 사용자 사례(Use Case) 전파
구축된 AI/Data 자산이 실무에서 활용되기 위해서는, 타 부서·타 계열사의 성공 사례가 실무자에게 직접적으로 와닿는 형태로 공유되어야 합니다. 단순 보고서·발표 형식이 아닌, 실무자가 ""내 업무에 이렇게 적용할 수 있겠다""고 즉시 체감할 수 있는 사례(업무 시나리오, 입력 데이터, 산출물, 재현 방법 포함)를 공유하고 직접 따라해 볼 수 있는 방식으로 전파되어야 한다고 봅니다. (설명이 아닌 유튜브처럼 하나씩 따라하기)

2. RAG / MCP 서버 인프라 제공
실무자 다수는 데이터 구축 자체보다 이미 구축된 데이터에 AI를 적용하여 효과적으로 활용하는 방향에 더 큰 관심을 두고 있습니다. 따라서 그룹 차원에서 RAG MCP 서버를 표준 인프라로 제공하여, 실무자가 별도 개발 없이 자신의 업무 도구(Copilot, Claude, Codex, 사내 챗봇 등)에서 직접 그룹 데이터를 호출, 활용할 수 있도록 지원하는 것이 가장 효과적이라고 봅니다. ""이런 방식으로 데이터를 끌어오니 잘 되더라""는 경험이 실무자 간 자발적으로 확산되면서, 사용 사례가 가장 빠르게 증폭될 것 같습니다.

3. 데이터 표준화 관련 실무 교육
실무자가 직접 데이터를 등록 및 활용하는 과정에서 표준 용어, 스키마, 메타데이터 작성 규칙을 자연스럽게 익힐 수 있는 기반 교육 프로그램이 필요합니다. 일방향 강의보다는 본인의 실제 업무 데이터를 표준에 맞춰 정제해보는 실습 중심 과정이 정착에 효과적이며, 표준화의 필요성을 실무자가 체감하는 가장 빠른 경로라고 생각됩니다."
	"4. 개발자(Developer) 대상 가이드 및 표준 제공
AI-Ready Data 체계에서 제공되는 상용 툴·플랫폼의 구성에 대한 정보가 아직 명확히 공유되지 않은 상황입니다. 향후 API 명세, 데이터 스키마, 인증·권한 모델, MCP 연동 방식, 샘플 코드 등 개발자 친화적 가이드가 함께 제공되어, 각 계열사·부서의 사내 개발 인력이 자체 응용 시스템을 구축할 수 있도록 지원이 필요합니다.

5. 데이터 라이프사이클 지원
데이터는 구축 시점이 종료가 아니며, 실무 수행 과정에서 새로운 인사이트가 발생할 때마다 추가 데이터에 대한 니즈가 지속적으로 도출됩니다. 따라서 데이터 확장 요청 → 표준화 검토 → 적재 → 재배포의 순환 프로세스가 상시 운영되어야 하며, 각 실무자가 직접 데이터 확장에 참여하고 즉시 결과를 확인할 수 있는 환경이 필요합니다."
