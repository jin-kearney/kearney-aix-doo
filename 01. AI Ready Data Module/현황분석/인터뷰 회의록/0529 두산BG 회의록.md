□	두산 전자BG 현업 인터뷰
	(커니) AI 과제 수행 시 어떤 Legacy System에 연동이 어려웠던 사례가 있었는지?
	(이정균 수석) 주로 진행하고 있는 과제가 제조 대상이기 때문에 MES나 ERP 시스템의 데이터를 많이 활용하고 있음. 그러다 보니까 모델을 개발할 때 CSV 파일 형태로 각각의 데이터를 필요에 의해 결합했는데 Production을 위해서는 시스템에 직접 접근해야 할 필요가 있음. 하지만 전자BG는 정보 접근이 제한적이고 시스템 운영을 DDI에서 하고 있기 때문에 운영 조직에 컨택을 해서 시스템의 DB 형태, 서버 스펙 등을 검토하는 게 시간이 오래 걸림.
	(커니) 실제 파일럿에서 Production으로 넘어가는 과정에서 어떤 어려움이 있었는지?
	(이정균 수석) 현재 AI 기반으로 자율 생산 체계를 구축하는 프로젝트를 진행 중임. 수립된 생산 계획을 가지고 실제 만들어야 될 제품을 AI가 자동으로 공정 세팅을 해서 양산하는 과제임. 과제를 수행하다 보니 MES 데이터와 ERP 데이터 등을 결합해야 했는데 해당 부분에서 어려움이 있었음.
	(커니) 파일럿을 진행할 때 현재 커니가 진행 중인 모듈2처럼 데이터를 받아서 진행한 것인지?
	(이정균 수석) 맞음.
	(커니) DB구조나 테이블 정의서가 없었던 문제가 있었는지?
	(이정균 수석) DDI가 실제 운영하고 있기 때문에 전자BG는 자체적으로 관리하고 있는 시스템별 DB 구조나 인터페이스 문서는 없음. 그러므로 직접 데이터를 가져와서 필요에 의해 조합하고 싶어도 DDI에서 잘 관리가 되지 않음. DDI에서도 운영 담당자가 계속 바뀌고 DDI에서는 뷰 테이블만 제공할 수 있음. 아마 DDI에는 테이블 정의서가 있을 것 같은데 업데이트 관리는 안 되고 있을 것임.
	(커니) DB현황 조사 시 가장 시간이 오래 걸린 단계는 무엇이었는지?
	(이정균 수석) 데이터 항목에 대해 파악하는 데 시간이 오래 걸렸음. 주로 시계열 데이터고, 이 항목이 AI/OE 팀에서 파악하고 있는 데이터가 맞는지 확인해야 했음. 
	(커니) 판단이 오래 걸렸던 이유가 무엇인지?
	(이정균 수석) AI/OE 팀에서 이해하고 있는 용어와 현업에서 사용하고 있는 용어가 다르고 그것에 대한 칼럼 정의가 모호했음. 그래서 메타데이터를 받으면 피처들을 리스트업해서 AI/OE 팀이 생각하는 게 맞는지 싱크를 맞추는 데 시간이 오래 걸림.
	(커니) 싱크는 현업과 협의해서 맞춘 건지?
	(이정균 수석) 모델링을 하기 위해서는 전체 프로세스를 알아야 하기 때문에 현업과 데이터 의미에 대해 싱크를 맞춤. 
	(커니) 싱크를 맞추는 작업이 프로젝트 총 기간 중 얼마를 차지했는지?
	(이정균 수석) 4분의 1정도였던 것 같음.
	(커니) 데이터를 연계할 때 직접 API를 활용했는지 혹은 DDI에 요청했는지?
	(이정균 수석) DDI의 시스템 운영자를 통해서 CSV 파일로 받아서 필요에 의해 조합했음.
	(커니) 그렇다면 Production 단계에서 연계를 하기 위해서는 DB에 직접 접근하거나 API가 필요한데 그 단계까지는 못 간 것인지? 
	(이정균 수석) 맞음. 전자BG AI/OE 팀은 DB 접근에 상당한 제한이 있음. 운영하는 조직 입장에서는 외부인에게 API를 열어준다는 것 자체가 리스크라고 판단하는 것 같음. 
	(커니) 현업에서 어떤 AI 과제가 필요하다는 드라이브가 있었기 때문에 과제화해서 Production으로 넘어가는 프로세스가 맞는지?
	(이정균 수석) 맞음. 그래서 DDI 담당자들이 AI 과제만을 위한 DB 구성을 따로 해 줌. 현재 Legacy 시스템 내의 설비 데이터 같은 경우는 파이프라인이 구축돼서 클라우드에 올라가고 있는 상태임. 전자BG는 설비의 공정 데이터가 중요하기 때문에 공정 데이터 수집을 위한 파이프라인을 구축하고 있고 이를 기반으로 재사용 가능한 체계까지 마련하고 있음.
	(커니) 체계를 마련하는 것은 전체적으로 마련하는 것인지? 혹은 과제에 한하여 마련하고 있는 것인지?
	(이정균 수석) 과제별로 Specific 하게 하는 것보다 제조 관점에서 필요한 데이터는 다 분석 해놓은 상태임. 그래서 공통으로 사용하는 데이터 파이프라인을 구축하고 있음.
	(커니) 공통으로 사용하는 파이프라인을 어떤 것인지?
	(이정균 수석) 메인은 품질 데이터였음. 데이터를 보다 보면 현업들이 다양한 아이디어를 제시할 수도 있고 개발을 하면서 개선할 수 있겠다는 아이디어를 낼 수 있어서 다양한 과제를 발굴할 수 있는 파이프라인을 구축하고 있음.
	(커니) 파이프라인에 들어가는 데이터를 정의하는 단계가 있는지? 아니면 그때마다 필요한 데이터를 찾아서 연결하는 형식인지?
	(이정균 수석) 파일럿 때는 필요한 데이터만 체크했고 현재는 Production 레벨이기 때문에 전체 현황에 필요한 데이터를 다 조사했음. 
	(커니) 어떤 방식으로 데이터를 조사했는지?
	(이정균 수석) 처음에는 질의 형태로 받았고, Production이 진행됨에 따라 과제를 공식화해서 공식적으로 문서를 받았음. DDI에 요청하는 것도 있었고 주로 설비를 만들다 보니까 설비 PC의 태그 정도나 업데이트 사이클 주기 같은 정보를 문서로 받음. 데이터가 정의되어 있는 것을 받아서 필요한 것을 매핑하는 방식이었음.
	(커니) 전달한 데이터와 현재 데이터가 동일하지 않은 이슈 같은 것이 발생하는 이유가 무엇인지?
	(이정균 수석) 담당자가 자주 바뀌고 관리 포인트가 없기 때문임. 
	(커니) 사전 질의서에 IT조직과의 협업이 의미하는 것이 DDI의 IT조직인지?
	(이정균 수석) 맞음. 전자BG에는 없음. DDI측의 운영에만 담당 IT조직이 있음.
	(커니) 다른 과제에서 또 다른 데이터 현황을 파악한 사례가 있는지?
	(이정균 수석) 모든 데이터 현황 파악은 현업 담당자를 통해서 진행했음. 현업 담당자들이 데이터에 접근하고 있고 그 데이터 접근을 통해 어떤 시스템에서 필요한 것들을 조합해서 쓰는지 일차적으로 확인함. 예를 들어서 AI모델을 개발하는 데 있어서 MES 데이터와 설비의 실시간 데이터가 필요했음. 왜냐하면 MES는 실적만 집계되는 데이터만 있었고 필요한 데이터는 제품을 만드는 기간 동안에 공정의 흐름을 확인하려고 했던 것임. 그래서 실시간 데이터의 시스템이 어디에 있는지를 현업 담당자를 통해서 알았고, MES는 DDI를 통해서 확인함.
	(커니) 그런 데이터들이 어떤 과제에서 필요했던 것인지?
	(이정균 수석) 앞서 말한 공정 제어 과제임. 그리고 다른 과제도 다 현업을 통해서 확인했음. AI/OE 팀은 시스템의 화면 단위의 데이터만 볼 수 있기 때문임.
	(커니) 필요한 데이터를 알고 있는 담당자나 운영팀을 찾는 데 어떤 애로사항이 있었는지?
	(이정균 수석) 실제 운영 담당자가 누구인지 몰랐음. 그래서 시스템을 사용하는 사람인 현업이 개선이 있을 때 연락하는 담당자를 알고 있어서 해당 커뮤니케이션 채널을 활용했음.
	(커니) 동일 데이터인데 용어가 다르거나 서로 이해하는 것이 달랐던 사례가 있는지?
	(이정균 수석) 같은 데이터를 가지고 생산에서 보는 관점이나 품질에서 보는 관점이 다 다름. 예를 들어서 제품을 생산하는데 공정 요건이 있을 때 생산의 입장에서는 최적의 조건이라고 생각하지만, 품질의 입장에서는 이렇게 만드는 것이 최적이 아닐 수도 있다고 생각함. 그래서 데이터 해석의 차이가 있음. 데이터 자체의 해석이 다른 것이 아니라 관점의 차이임.
	(커니) 데이터 설명을 특정 담당자 경험에 의존하는 경우가 많은지?
	(이정균 수석) 많음.
	(커니) 데이터의 의미를 확인하는 것 때문에 과제를 바꾼 사례가 있는지?
	(이정균 수석) 데이터를 확인한 다음에 과제를 만들었음.
	(커니) 데이터를 확인하는 과정은 총 프로젝트 기간 중 얼마나 걸렸는지?
	(이정균 수석) 전체 과제가 3달이라고 하면 3분의 1정도는 데이터 확인을 했음. 
	(커니) MES를 주로 사용하는데 어떤 것이 수기로 관리되고 있는 것인지?
	(이정균 수석) 모듈 2 온톨로지를 보면 알 수 있을 텐데 품질 쪽은 거의 다 엑셀로 관리하고 있고 SCM의 조직들도 각 시스템을 다운 받아서 본인만의 인사이트를 넣어서 관리하는 엑셀이 있음.
	(커니) SCM도 다른 계열사를 통해서 듣기론 PI를 진행하고 있다고 들었는데 어떤 작업을 하고 있는 것인지?
	(이정균 수석) 전자BG AI/OE 팀에서 진행하고 있는 스마트플랜 과제는 대부분 엑셀로 계획을 수립하고 있음. 그 사유는 데이터가 없어서가 아니라 현재 비즈니스 환경이랑 시스템 컨셉이랑 달라서 엑셀을 활용하는 경우임.
	(커니) 비즈니스 환경과 시스템 컨셉이 맞지 않다는 것이 어떤 의미인지?
	(이정균 수석) 예전에 만들던 제품의 형태, 주문의 형태와 현재의 형태가 다른데 시스템에는 그게 반영이 안 되어 있어서 PI를 다시 하는 것임. 그래서 시스템 제약 때문에 엑셀을 많이 활용하고 있음.
	(커니) 현업에서 시스템보다 엑셀 데이터를 활용하는 비중이 어떻게 되는지?
	(이정균 수석) 대부분임. 품질 데이터 과제를 진행하면서 현업에 물어보니 자재에 대한 출하 검사 같은 데이터는 SAP 데이터보다 엑셀 데이터가 더 신뢰할 만하다고 답변받았음.
	(커니) 데이터 탐색, 이해, 접근 과정을 위한 특정 담당자가 있는 것인지?
	(이정균 수석) 없음. 현업별로 시스템을 많이 쓰는 사람을 수소문해서 하는 형태임.
	(커니) 정합성 판단에 시간이 많이 소요된다고 답변했는데 어떤 내용인지?
	(이정균 수석) 사람이 입력하는 데이터에 대해 정합성 문제가 존재함. 
	(커니) 시스템의 센서에 대부분 입력되는 것이 아닌지? 사람이 입력해야 하는 것인지?
	(이정균 수석) 센서에 입력되는 것들은 중요 인자가 아닌 경우가 존재하고 제품의 특성 같은 것들은 사람이 입력하는 경우가 많음. 수치 형태로 시스템에 입력하는 것임. MES 같은 경우에 입력하는 시점에 따라서 시스템에 저장되는 데이터가 다름. 예를 들어서 제품 하나를 만드는 데 2시간이 걸린다고 했을 때 MES에는 한순간만 저장이 되기 때문에 이 데이터가 기록되는 시점이 초반인지 중반인지 알 수 없음.
	(커니) 왜 그렇게 한 시점만 기록되는 것인지?
	(이정균 수석) MES 컨셉 자체가 실적 처리를 목적으로 만들어졌고 오래된 시스템이기 때문인 것 같음.
	(커니) 기록되는 나머지 필요한 데이터는 아예 없는 상태인지?
	(이정균 수석) 따로 정리되고 그걸 옮기는 형태임. 제품의 Pass와 Fail을 판단할 수 있는 주요 데이터는 현장에 수기 관리하고 있는 파일에 기재를 하고 이를 사람이 시스템에 입력하는 프로세스임. 그러다 보니 사람이 실수할 수가 있어서 이것의 정합성을 판단하는 시간이 많이 소요됨.
	(커니) 정합성에 대한 판단 그 자체는 어떻게 하는지?
	(이정균 수석) 과거 데이터 이력을 보고 정합성을 아웃라이어라고 판단하고 이를 현업들과 커뮤니케이션해서 규명함.
	(커니) 현업과 소통해서 그 데이터 값을 사후적으로 돌릴 수 있는 것은 아닐 것 같은데 어떤 식으로 정합성을 맞추기 위해 처리하는지?
	(이정균 수석) 그래서 따로 파이프라인을 만들어야 한다고 결론지었음. 정합성 처리를 사전에 조치해서 감지하는 것이 안 될 것 같아서 사람의 개입을 최소화하는 방향으로 진행 중임.
	(커니) 그것은 근본적인 해결책인 것 같은데 혹시 과제를 진행하면서 아웃라이어 데이터를 어떻게 처리했는지?
	(이정균 수석) 제거를 하는 경우도 있고 유사한 값을 판단해서 입력하는 경우도 있었음.
	(커니) 특정하게 어떤 시스템에서 정합성 문제가 더 자주 발생하는지?
	(이정균 수석) MES 시스템임.
	(커니) 동일 데이터라도 시스템별 값이 다르게 관리되는 사례가 있는지?
	(이정균 수석) 데이터를 연결하는 인터페이스가 되어 있어서 그런 경우는 많지 않음.
	(커니) 데이터 정합성 판단 시간이 다소 소요됐던 사례가 있는지? 혹은 아예 못 믿겠다고 판단한 사례가 있는지?
	(이정균 수석) 현장의 기술직 엔지니어가 작업하는 경우가 많으므로 데이터 균질성 확보가 어려웠음. 아예 못 쓸 정도는 거의 없었고 최대한 데이터를 활용하고자 했음.
	(커니) 데이터 생성·변환 이력을 추적할 수 있는 체계가 있는지?
	(이정균 수석) 현재 구축하고 있는 과제의 컨셉이 엑셀 데이터를 최대한 유사한 형태로 플랫폼 내에서 사람들이 관리하게 하는 것이고 누가 데이터를 생성하고 변경했는지 추적이 가능하게 구현 중임. 자체적으로 코드 베이스로 기능 구현을 해서 수행 중임.
	(커니) 엑셀을 자산화하는 방식이 어떤 것인지? 
	(이정균 수석) 현재 엑셀로 관리하고 있는 데이터는 주로 사람이 측정하고 모니터링하는 데이터들임. 그런 것들을 입력하게끔 하는 기능을 코드로 만드는 것임.
	(커니) 데이터 추출을 위해서 DDI 조직의 확인이 필요하다는 것이 운영팀을 말하는 게 맞는지?
	(이정균 수석) 맞음. 현업 담당자를 확인하고 메일이나 시스템 요청을 하고 있음. 그러면 데이터를 메일로 전달받는 형태임.
	(커니) 데이터 현황 파악이나 추출 가능 여부를 확인하는 데 얼마나 시간이 소요되는지?
	(이정균 수석) 현황 파악은 레거시 화면을 기준으로 확인하고 절차대로 진행했을 때 빠르면 다음 날 혹은 3~4일 정도 소요됨. 
	(커니) AI과제를 하면서 전처리 하는 것이 유난히 어려웠던 유형이 있는지?
	(이정균 수석) 최근에 OCR 파이프라인을 구축했는데 성적서의 형태가 PDF, 이미지, 엑셀 등 다양하게 존재했음. 앞서 언급한 수기 엑셀로 데이터를 관리하고 이를 시스템에 수작업으로 입력해야 했던 것을 AI로 대체할 수 있도록 하기 위함임. 성적서는 고객사로부터 받고 실험 장비에서 워드 형식의 보고서 형태로 나옴. 그래서 이를 사람이 입력하게 되면 시간도 많이 걸리고 정합성도 확인할 수 없어서 이를 OCR로 자동화하고자 과제를 진행했고 완료함. 
	(커니) 수작업으로 데이터를 어디에 입력하는 것인지?
	(이정균 수석) 별도로 플랫폼을 만들어서 업로드함. 
	(커니) 자체 개발을 하는 과정에서 DDI와 협업이 이루어지는지?
	(이정균 수석) 파이프라인은 전자BG AI/OE 팀에서 설계하고 인프라에 접속하는 것은 DDI를 거쳐야 하므로 구축은 DDI가 진행함.
	(커니) OCR의 모델 결정은 어떤 팀에서 결정하는 것인지?
	(이정균 수석) 전자BG AI/OE팀에서 모델과 파이프라인을 선정해서 협력사와 함께 검증함. 
	(커니) 데이터 구조나 형식이 시스템별로 달라 추가 가공이 필요한 경우가 있었는지?
	(이정균 수석) 문서 형태가 다양하기 때문에 이를 커버할 수 있는 파이프라인을 구축함. 관리하고 있는 정보를 프롬프트에 넣어서 방식과 양식이 바뀌어도 커버할 수 있는 프롬프트를 구축했음. 현재 운영을 하기 직전인데 다양한 케이스에 대해서 연결은 문제가 없음.
	(커니) 현업과 협업이 잘 안 이뤄지는 상황이 있는지?
	(이정균 수석) 초기에는 잘 안 이뤄지는 경향이 있었는데 솔루션의 효과를 보니까 오히려 더 현업에서 오픈을 하게 됨.
	(커니) AI 활용 문화 확대를 위해 가장 먼저 필요한 것은 무엇이라고 생각하는지?
	(이정균 수석) 사내에 어떤 데이터가 있는지 이해하는 것이 가장 중요함. 이를 위해서는 조직이나 플랫폼의 표준이 필요하다고 생각함. 현재 전체를 관리하고 있는 조직이 사내에 없어서 DDI에 의존하게 됨.
	(커니) 사전 질의서에 AI과제가 일부는 자세하게 적혀 있고 일부는 비워져 있는데 어떤 것인지?
	(이정균 수석) 장표 상 1~4번은 본인이 진행한 과제이고, DDI가 적어준 14번을 제외한 12~16번은 1~4번과 동일한 것임. 나머지 5~11번은 전자BG AI/OE 팀에서 진행한 것들임.
	(커니) AI 공정 제어 시스템 구축 과제에서 생산 실적 데이터만 존재하고 생산 중 공정 데이터가 없다고 답변한 것의 의미가 무엇인지?
	(이정균 수석) 앞서 답변한 MES가 특정 시점의 데이터만 저장해서 저장되지 않는 데이터가 있다는 것임.
	(커니) 필요했던 공정 데이터는 생산 중에 있는 공정 성적서 같은 것들인지?
	(이정균 수석) 공정 성적서도 있고 센서 같은 데이터는 현업과 추가하면서 과제를 진행하고 있음.
	(커니) Lot 기준 추적 시 연결이 어려운 데이터나 시스템이 있는지?
	(이정균 수석) 커니에서 진행 중인 품질 과제도 시스템에 있는 것 외에 사람이 엑셀로 관리하는 것을 어떻게 반영할지 고민을 하고 있을 것 같음. 왜냐하면 자체적으로 Lot을 관리하는 기준이 있기 때문임. 그래서 아예 엑셀을 플랫폼 기능에 넣었음.
	(커니) 제조 Issue 대응 시스템은 어떤 것인지?
	(이정균 수석) 하반기부터 진행할 과제이고 품질 쪽과 유사함. 품질 관련된 것은 지금 하고 있고 이외에도 어떤 제품을 만들었을 때 최적의 수치를 생성해 내는 에이전트를 개발하고 있음.
	(커니) 현재 어떤 단계인지?
	(이정균 수석) 과제 정의는 되어 있고 현재 데이터를 확보하고 있음.
	(커니) 데이터 확보는 어디서 하고 있는지?
	(이정균 수석) 현업 담당자를 통해서 하고 있고 타임라인이 다 구축되어 있음.
	(커니) 모듈 2에서 하고 있는 부분과 겹치는 부분이 있을 것 같은데 맞는지?
	(이정균 수석) 유사함. 아마 데이터 출처도 같을 것 같음. 하지만 그 데이터를 품질 관점에서 보는지, 공정 최적화 관점에서 보는지 차이임.
	(커니) 공정 최적화 관련해서는 과제 정의와 구체화가 다 된 상태인지?
	(이정균 수석) 맞음. 앞서 AI 모델을 만들었기 때문에 프로세스와 내부적으로 데이터가 어떤 것이 있는지 다 알고 있음. 여기에 추가로 암묵지적인 것이 뭐가 있는지 현업과 공감대가 형성되어있는 상태임. 그래서 이를 구체화하기 위한 논의가 되어있음. 아마 커니에서 하고 있는 과제의 결과물도 이 과제에 활용할 수 있을 것 같음.
	(커니) 여기서 말하는 정형 데이터는 시스템에 올라가 있는 것이고 비정형 데이터는 엑셀 같은 것들인지?
	(이정균 수석) 맞음. 
	(커니) 조치 보고서는 어떻게 관리하고 있는지?
	(이정균 수석) 8D 리포트도 포함되는데 공정 단에서 관리하고 있는 문서를 파악해 놓은 상태임.
	(커니) 조치 보고서 역시 형식이 달라서 전처리가 필요한지?
	(이정균 수석) 맞음.
	(커니) 데이터 로딩에 장시간이 소요되는 이유가 무엇인지?
	(이정균 수석) 생산계획을 수행하기 위해서는 계획에 필요한 데이터가 있는데 데이터를 가져오는 것 자체가 주문의 양을 다 인터페이스 해야 하고 주분별로 내부적으로 배치하는 기준이 있기 때문에 중간에 처리되는 로직이 많음. 그래서 데이터양도 많고 처리하는 로직도 많으므로 시간이 오래 걸림. 그래서 해당 Pain Point를 해결하기 위해 PI를 마무리한 상태임.
	(커니) 그럼 Pain Point는 데이터가 분절돼 있어서 생기는 문제인지?
	(이정균 수석) 정형화된 데이터를 가지고 생산계획을 짜기 때문에 분절 이슈는 아니고 비즈니스와 안 맞는 시스템이 운영되기 때문에 중간 처리 로직을 넣어서 그런 것임.
	(커니) 11개의 과제 모두 DDI와 함께하고 있는지?
	(이정균 수석) 맞음. 하지만 DDI의 참여도 수준은 다 다름. 6번을 제외한 5~11번은 DDI가 모델까지 다 하는 컨셉이고 1~4번은 AI/OE 팀에서 모델도 만들고 파이프라인을 설계한 과제임.
