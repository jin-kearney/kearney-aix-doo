□	1. 데이터 카탈로그
	(커니) 조직 내에 데이터 카탈로그가 있는지?
	(정성문 수석) 데이터 카탈로그 없음. 타 조직의 데이터가 필요한 경우에는 감각적으로 어떤 데이터가 누구에게 있는지 파악하여 요청함. 또한 매년 조직에서 관리하는 DB가 바뀌기 때문에 데이터의 위치와 칼럼 자체가 달라지는 경우가 있음. 그러므로 작년에 파악한 데이터 위치와 올해 저장된 데이터 위치가 다를 수 있음. 데이터 DB가 업데이트될 때 데이터 카탈로그도 같이 업데이트 되어야 하는데 그렇지 못함.
	(커니) 어떤 맥락에서 데이터 칼럼 자체가 바뀌게 되는지?
	(정성문 수석) 예를 들어 과거에 원자력 위변조 사건이 발생하기 전에는 위변조 관련 데이터 없었는데 사건 발생 후 위변조 검사 항목을 데이터화 해서 관리하게 됨. 이런 식으로 데이터 칼럼이 생겼다가 더 이상 관리할 필요가 없다고 판단이 되면 데이터 입력을 하지 않음. 실무에서는 데이터 DB를 유연하게 관리하려고 하는 편임.
	(두산지주) 과거 AI/Data 과제를 진행할 때 데이터 칼럼을 찾고 데이터를 확인하는 데 시간 소요가 많이 된 것으로 알고 있고 담당자가 데이터 칼럼을 캡쳐해서 어디 시스템에 어떤 DB랑 연결되었는지 골라냈다고 들었음. 실제로 그렇게 해야 할 과제가 많이 발생했는지?
	(정성문 수석) 많이 발생함. 예를 들어 DB구조가 parent-child 구조일 경우에는 한땀 한땀 찾아야하는 케이스가 많았음.
	(두산지주) 파워서비스 BG에서도 데이터 카탈로그 관련 이슈나 pain point가 있었는지?
	(이성민 수석) 시계열 데이터 (운전 데이터)를 많이 모으고 있고 운전 데이터를 적재하는 시스템을 만들고 있음. 해당 과정에서 문제점은 현장에서 오는 데이터 태그가 같은 데이터인데도 다르게 들어오는 경우가 있음. 그럴 경우에 사람이 일일이 같은 데이터라고 매핑을 해서 수정을 하는데 그 다음에 현장에서 데이터가 들어올 때 또 다른 태그로 들어오게 됨. 도메인 지식이 있는 사람은 이를 판단하고 정제할 수 있음. 이를 해결하기 위해 비슷한 description의 데이터는 같은 데이터로 묶는 식의 정제를 할 수 있는 스크립트를 짜서 실행하고 있지만 불편하고 어려움. 또한 프로젝트가 달라지면 아이디 값 등이 달라져 쓸 수 없음.
	(커니) 해당 문제는 데이터 소스가 달라져서 발생하는 문제인지?
	(이성민 수석) 같은 데이터여도 계측기 자체의 데이터 소스를 받아올 때도 있고 제어 시스템의 인풋/아웃풋일 때도 있음. 또한 중복된 데이터가 들어올 때도 있고 평균 값을 낸 데이터, 원본 데이터가 들어올 때도 있음.
	(커니) 같은 데이터 소스인데 왜 데이터를 수집할 때마다 다른 방식/시스템에서 수집을 하는 것인지?
	(이성민 수석) 운전 데이터는 보안데이터이기 때문에 실시간으로 받을 수 있는 데이터는 제한적이며 현장 인력들이 CSV파일로 받고 정제해서 전달하기 때문임
	(커니) 그렇다면 현장 인력에게 CSV를 만들 때 instruction이나 role를 지정하지는 않았는지?
	(이성민 수석) 그렇게 못함. 데이터는 SCADA나 히스토리안을 통해서 받아오는데 그 프로그램에서 아웃풋을 내보내는 게 정해져 있어 바꾸기 위해서는 제어 시스템을 새로 개발해야함.
	(커니) 그럼 데이터를 뽑을 때 SCADA로만 뽑으면 해결되는 것이 아닌지?
	(이성민 수석) 같은 데이터를 보더라도 제어 시스템에서 가져다 쓰는게 여러 개이므로 a 신호 값을 받을 수도 있고 b 신호 값을 받을 수도 있음
	(두산지주) 데이터들의 차이가 시간 단위라든지 다 다른가?
	(이성민 수석) 원본 데이터는 다르지 않음. 타임 스탬프를 다르게 가져다주는 경우가 제일 많음.
	(두산지주) 구축하고 있는 것으로 어느 정도 선까지 해결되고 있는지?
	(이성민 수석) 사내에서 구축 가능한 것은 리얼 타임 데이터는 아니고 프로메테우스 솔루션으로 도입하려고 했는데 비용 문제 때문에 안됨. 일단 원본 엑셀이나 CSV파일 raw 파일 관리하는 것으로 DB 만들 예정임. 거기에 오픈 소스로 사용자들이 데이터 확인할 수 있는 Apache Super Set을 붙이는 방식을 구축하려고 생각하고 있음. 일단은 어떤 데이터가 있는지 확인할 수 있게 대시보드 정도 만들 예정이고 API는 별도 개발해서 필요한 데이터를 추출해서 분석할 수 있도록 시스템 개발 중임.
	(두산지주) 데이터 분석팀은 데이터를 현장bg보다는 좀 더 체계가 있을 것으로 예상되는데 관련 내용 공유 가능한지?
	(김용화 수석) 저희 팀은 다이브를 가지고 있고 기반으로 RMS 서비스하고 해서 터빈 쪽 운전 데이터를 매일 단위로 쌓고 있음. 다만 이를 제어에 활용하는 수준은 아니고 SCADA에서 활용하는 수준임. 매일, 매주, 매월 발주처의 데이터를 기반으로 분석 및 리포팅하는 기능임. 데이터에 대해 적재하거나 처리하는 과정에 대해서는 부족한 점 없음.

□	4. 데이터 전처리
	(커니) 전처리 과정 중 어떤 반정형/비정형 데이터를 정규화하는 데 시간이 많이 걸렸는지? 파서 같은 에너빌리티 내부에서 사용하는 방법론이 있는지?
	(김광일 수석) 레이아웃을 만들기 위해 2D치수를 하나하나 측정해서 3D로 옮겨서 캐드 파일로 만드는데 이 과정이 시간이 오래 걸리며 이 레이아웃을 추출해서 AI에게 학습시키는 과정이 어려웠음
	(커니) 일반적인 데이터 전처리는 세세하게 다룰 예정인데 에너빌리티 혹은 본인이 수행한 과제에서 특별히 조금 더 검토를 희망하는 부분이 있는지?
	(이성민 수석) 에너빌리티 내부에서 RAG을 직접 만들어 쓰는 사람이 많음. 오래된 문서 중에 스캔 퀄리티가 낮은 파일이 많은데 해당 경우에는 OCR이 안됨. 그래서 담당자들이 수기로 워드 파일을 만들어 작업했음. 또한 role-based 설계 tool을 만들려고 시도했는데 설계절차서와 같은 파일은 옛날 pdf파일들이라 이를 학습시키지 못했음. 즉 옛날 pdf 파일은 전처리를 거의 못하고 있음.
	(커니) 문서를 전처리 할 때 정해진 파서 같은 tool은 없고 개별적으로 하고 있는 것인지?
	(이성민 수석) 맞음. 계약서 입창 당시에 오는 문서는 PI팀에서 만들어준 시스템을 사용할 수 있지만, 모든 사용자에게 오픈해주지 않고 신청을 받아서 일부에게만 제공함. PDF와 워드 같은 파일 형식도 지원함. 계약서 전처리 후 계약서에 어떤 조항이 있는지 검토하는 프로그램임.
	(정용민 수석) 계약서 검토 시스템 관련해서 추가 문의사항이 있다면 정리해서 전달주면 담당자 확인해서 설명 가능함. 
	(커니) Role-based 설계 tool을 만들었다고 했는데 구체적으로 어떤 것인지 설명가능한지?
	(이성민 수석) 현재 설계 절차가 왔다갔다 하기 때문에 최대한 role을 만들어서 입력값과 출력값이 일정할 수 있도록 자동화하는 프로그램임.
	(커니) 그럼 해당 프로그램에서 어떤 맥락에서 데이터 전처리를 해야하는 것인지?
	(이성민 수석) 설계 데이터는 전부 문서로 전달받기 때문에 사람이 일일이 값을 보고 입력하는 과정을 없애기 위함. 만약 설계 데이터가 DB에 저장되면 자동으로 불러올 수 있도록 하는 것을 생각 중임. 또한 현장 시계열 데이터를 개별적으로 전처리해서 머신러닝을 돌리고 있기 때문에 AI를 활용해서 운전 데이터, 운전 설계, 지침 등을 판단할 수 있도록 하고 싶음. 
	(커니) 개별적으로 전처리를 진행한다고 하셨는데 담당자마자 전처리 과정이 상이한지?
	(이성민 수석) 전처리 기준이나 방법이 없고 본인의 목적에 맞게 전처리를 하기 때문에 모두 상이함.

□	6. 온톨로지
	(커니) 온톨로지를 구축하는 과정에서 entity 정의 방법이나 범위 등 어려웠던 점이 있는지?
	(김용화 수석) 데이터 분석팀에서 작년부터 올해 1월까지 가스 터빈과 제어로직에 대해서 온톨로지를 만들었고 어려웠던 점은 도면 PDF 파일을 딥러닝 통해서 추출하는데 오류가 많이 발생했으며 수작업이 많이 필요했음. 체계 관련해서는 이미 설계 쪽에서 체계를 갖춘 채로 도서를 만들었기 때문에 이를 충실하게 반영하는 형태로 구성하고자 했음. 이에 필요한 데이터로 제어 센서 값, 설정 값은 파편화 된 자료를 한 부서에서 모아서 온톨로지로 구성했음. 이를 LLM을 통해서 질의 응답하는 기능까지 구축했음. 현재는 모델 같은 이슈 사항이 있어서 튜닝 및 테스트 작업 진행 중임.
	(김용화 수석) 도면 내에 있는 선을 어떻게 디텍팅해서 정확하게 객체들 간의 연결 관계를 추출해내는 것은 아직 오픈 소스 모델로 작업하기 어렵다고 느낌.
	(김용화 수석) entity에 대한 정의는 명확하게 되어 있어 도면이 형성이 되는 건데 entity의 관계를 추출하는 과정에서 DWG 파일에서 바로 데이터로 파싱할 수 없는 방법이 없음.
	(커니) 원본이 틀려서 발생한 문제인지? 혹은 원본을 파싱하면서 AI에 맡겼기 때문에 겹쳐 있는 선을 인식 못해서 사람이 후작업을 하는게 어려웠던 것인지?
	(김용화 수석) 도면 자체는 사람이 인식하면 더 이상 문제되지 않음. 근데 도면을 AI가 인식하기 위해선 정확하게 선으로 연결이 되어야 하지만 살짝 띄어져있거나 하면 AI는 다르게 인식함. 또한 사람이 도면 작업을 하다 보니 글자나 방향 같은 것이 달라질 수 있어 오류가 생길 수 있음. 그래서 100% 자동화하는데 어려움이 있음.
	(커니) 온톨로지 구축에 중요한 점은 entity의 정의 및 속성이라고 생각했는데 그런 것들은 어느 정도 도면 상에 명확하게 정의되어 있고 도면을 AI로 전처리하는 과정에 더 어려움이 있는 것이 맞는지?

□	7. 데이터 품질 권한관리
	(커니) 데이터 품질이나 권한을 관리하는 프로세스가 있는지?
	(이성민 수석) 없음.
	(정성문 수석) 품질 검사는 외부의 용역을 활용해서 진행하고 용역별로 각자 다른 파일 (엑셀, pdf 등)을 두산에 주고 시스템에는 참조 파일만 옮기게 됨. 그러면 해당 문서를 파싱해서 필요한 필드에 채워 넣는데 결과가 너무 많아서 일일이 검증할 수 없음. 이게 결과적으로 데이터 품질이 떨어지는 핵심 요인이 됨.
	(두산지주) 외주 용역 업체에 어떤 체계 혹은 게이트를 만들어 그것을 패스해야 리포트가 통과하게 만드는 방식을 만들 수는 없는지?
	(정성문 수석) 두산이랑만 계약을 한 것이 아니기 때문에 그들의 문서가 달라지게 되면 그들이 계약한 회사들에게 컴플레인이 들어와서 힘듦. 계약에 해당 내용을 반영하고 싶어도 이미 계약했기 때문에 신규 갱신까지는 바꿀 수 없음.

	(김광일 수석) 도면 관리 시스템이 있어서 최신 버전을 확인할 수는 있지만 특정 버전이 어떤 프로젝트에 쓰였는지와 같은 부분은 관리가 안됨. 그래서 이력 관리가 되기는 하지만 직접 들어가서 확인을 해야함.
