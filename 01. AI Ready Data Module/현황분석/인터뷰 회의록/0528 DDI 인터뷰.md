□	두산 DDI 현업 인터뷰
	(커니) DDI 사전 질의서는 누가 작성한 것인지?
	(박승현 수석) 이은광 팀장이 작성해서 박승현 수석이 전달한 것임. 사전 질의서에 적은 25개의 과제는 현재 수행한 것이 아니라 DDI 내에서 진행했던 프로젝트 전체를 나열한 것임.
	(커니) 현업의 눈높이 조정이라는 답변이 어떤 사례인지?
	(이은광 팀장) DDI 자체에서 과제를 수행하는 것은 어려움이 없지만 현업 내에서 눈높이가 다른 점이 어려웠던 것 같음. 여기서 말하는 현업은 임원들을 말한 것인데 임원들의 입장에서 AI 과제라고 하면 굉장히 광범위하고 뭐든 해낼 수 있는 도구라고 생각해 기대 수준이 매우 높음. 하지만 실제로 AI 과제를 수행하기 위해서는 데이터 준비가 필요한데 이를 간과하는 것 같음. AI 과제를 수행하기 위해 AI에 필요한 데이터 준비와 AI의 한계 등을 아는 마인드셋이 필요함.
	(커니) 그렇다면 실제 프로젝트를 진행하는 PM은 어떤지?
	(이은광 팀장) PM은 프로젝트를 직접 수행하기 때문에 그런 마인드셋이 주입되는 것 같음. 하지만 PM들도 임원들의 동의를 받아야 하므로 어려운 것 같음.
	(커니) 데이터 관점에서 과제를 수행하는데 어려움이 있던 사례가 있는지?
	(이은광 팀장) 여러 스텝에서 발생하는데 첫 번째로 과제 정의가 안 됨. 과제 정의는 AI로 무엇을 만드는 것으로 끝나는 것이 아니라 데이터 추출 같은 과정이 필요한데 결과만 요구함. 그래서 문제 정의가 명확하지 않음. 이런 문제는 프로젝트화 되기 전에 수정 가능한 부분이긴 함. 두 번째로는 데이터의 준비 수준임. 계열사별로 준비 수준이 다 다르므로 데이터 준비도의 차이가 있음. 예를 들어서 전자BG는 ERP 등의 시스템을 잘 갖추고 있는 편이기 때문에 정형 데이터를 쉽게 얻을 수 있지만 테스나 같은 경우에는 ERP 같은 시스템을 잘 갖추고 있지 않기 때문에 데이터 수집에 어려움이 있음.
	(커니) 데이터 수집이 문제가 된다고 하는 것은 데이터가 설비 로컬PC에 있기 때문인지?
	(이은광 팀장) 맞음. 데이터가 어떤 서버에 올라와 있는데 그 서버를 연결하는 것이 문제가 되기도 함. SAP 같은 곳에서 정형 데이터를 가져오는 것에 대한 문제는 없는데 그 외에 설비로부터 주고받는 텍스트 데이터에 대한 준비 수준은 거의 없음.
	(커니) 정형데이터의 경우에 테이블 정의서 같은 것이 없어서 애로사항이 있다고 들었는데 맞는지?
	(이은광 팀장) 상황에 따라 다를 것 같은데 새로 만든 시스템의 경우에는 잘돼 있을 것 같음. 
	(박승현 수석) 레거시 시스템의 경우에는 10년 이상 장기간 운영되고 있는데 그런 경우에는 DB나 스키마가 업데이트가 되지 않거나 하고 있음. 그래서 AI CoE팀에 최신화를 요청해서 협의하는 과정임.
	(커니) 운영팀에서 어떤 데이터가 어떻게 구성되어 있는지 알고 있는지?
	(이은광 팀장) 시스템적인 것은 DDI의 담당자가 오래 운영했기 때문에 더 잘 아는 부분이 있음. 그래서 해당 부분은 문제가 크게 되지 않는데 특정 데이터가 특정 누군가에게 있는지 알아내는 것이 어려움. 그래서 시스템 담당자가 현업 담당자보다 많이 알고 있는 경우가 있고 현업 담당자가 바뀌는 주기가 시스템 담당자가 바뀌는 주기보다 더 빠름. 
	(커니) 프로젝트를 진행하기 전에 기대치를 조정하는 Gate가 있는지?
	(이은광 팀장) 항상 있지는 않음.
	(커니) 현업이 필요한 데이터를 먼저 정의하고 DDI에 데이터를 찾아달라고 요청하는 경우가 있는지?
	(이은광 팀장) 그런 경우도 있고 프로젝트에 따라 다름. 조직 내부에서만 진행하는 과제일 경우에는 어떤 목적으로 데이터를 요청하는지 모름.
	(커니) 현업이 알고 있는 데이터와 시스템 데이터 간 차이가 발생하는 것은 어떤 사례인지?
	(이은광 팀장) 데이터의 수치 값이나 디스크립션이 다른 게 아님. 현업이 말하는 데이터가 특정 화면에 있는 데이터를 요청하는 경우가 있는데 화면에 있는 데이터는 사실 여러 개의 테이블로 구성되어 있어서 단순히 데이터를 끌고 와서 사용할 수 없음. 예를 들어 SAP에서 재고 데이터를 볼 때 전자BG에서 매번 직접 들어가서 확인하기가 불편하므로 전자BG의 에이전트 시스템을 통해서 볼 수 있게 하고 싶다고 함. 그래서 현업이 SAP 화면을 캡처해서 DDI에 요청하게 되면 사실상 해당 화면은 수많은 테이블로 이루어져 있는 데이터기 때문에 실시간으로 가져올 수가 없음.
	(커니) SAP 테이블에서 데이터의 위치는 운영팀이 관리하고 있는지?
	(이은광 팀장) SAP 담당자가 알고 있음.
	(커니) 데이터의 위치 같은 것을 검색할 수 있는 기능이 있는지?
	(이은광 팀장) 쉽지 않음. SAP의 경우에는 자기들 만의 솔루션이 있어서 그걸 사용해야 함. 예를 들어서 창고 하나의 데이터는 10개 정도의 테이블로 구성이 되어 있는데 이거를 엮어서 검색할 수 있게 만드는 것은 사실상 불가능하므로 SAP의 솔루션을 이용해야 함.
	(커니) 데이터 정의서 같은 것들이 자료로 저장돼 있는지?
	(이은광 팀장) 시스템별로 관리하는 것이 있을 것 같음. 담당자가 아니기 때문에 잘 모르겠음.
	(커니) 데이터 취합 시 데이터 소스가 다르거나 전처리에 필요한 Key가 안 맞거나 같은 어려움이 있었는지?
	(이은광 팀장) 비정형 데이터의 경우에는 Key를 맞출 수 없으니 그런 어려움은 없었음. ERP는 시스템 담당자와 커뮤니케이션을 통해 정형 데이터를 취합할 수 있음. 최근에는 비정형 데이터와 정형 데이터를 연계하는 것에 대한 니즈가 많은 것 같음. 비정형 데이터에서 원하는 것을 에이전트로 추출해서 답변받고 기존의 정형 데이터를 불러오는 것을 요구함.
	(박승현 수석) 대부분의 시스템을 DDI에서 운영하기 때문에 담당자만 찾으면 큰 어려움은 없음. 시스템 담당자는 계속해서 프로젝트를 같이 진행한 사람이면 쉽게 찾지만 처음 프로젝트를 하는 사람일 경우엔 커넥션을 찾기 어려움. SAP의 경우에는 모듈별로 담당자가 다르므로 담당자가 매우 많음.
	(커니) SAP를 관리하는 조직의 구성이 어떻게 되는지?
	(박승현 수석) 모듈별로 팀이 묶여 있고 그 안에서도 계열사별로 담당자가 나뉘는 경우가 있음.
	(이은광 팀장) SAP 생태계가 작은 생태계가 아니기 때문에 팀 하나로 묶을 수 있는 것은 아니고 SCM, 생산 등으로, 영역별로 묶어서 그 안에 세부 담당자가 있는 구조임. 그래서 계열사별로 꼭 있는 것은 아니고 에너빌리티 같은 경우는 SAP를 안 쓰기 때문에 또 따로 있음.
	(커니) 담당자 의존 방식에서 문제가 발생한 사례가 있는지?
	(이은광 팀장) 프로젝트를 진행하면서 그런 경우는 없었지만 다른 사람들이 진행하면서 그랬을 경우가 있을 것 같음.
	(커니) 어디에 있는 비정형 데이터가 수집하기 어려웠는지?
	(이은광 팀장) 어떤 프로젝트인지에 따라 달라짐.
	(커니) 비정형 데이터에서 불필요한 정보의 기준과 추출 대상 항목을 어떻게 정의하는지?
	(이은광 팀장) 정의하기 어려움. 예를 들면 특정 프로젝트에서 이메일이 필요한데 본인이 영업 쪽 담당자라면 고객 정보 같은 영업 관련 데이터 추출을 원할 텐데 시스템 담당자는 그것을 알 수가 없음. 또한 본인뿐만 아니라 비슷한 데이터를 요구하는 사람들에게서 공통적인 정보를 추출해야 하므로 정의할 수 없음.
	(커니) 비정형 데이터의 품질 관리 시 가장 먼저 선행돼야 하는 것이 무엇인지?
	(이은광 팀장) 문제에 대한 정의를 명확하게 하는 것. 예를 들면 영업 담당자가 고객에게 줬던 인사이트를 추출한다고 했을 때 인사이트는 무엇이고 대상 고객은 누구인지 명확히 정의가 필요함.
	(커니) 각 AI 조직에서 각각의 과제를 실제로 어떤 식으로 진행했는지?
	(박승현 수석) AI CoE팀 같은 경우는 3월에 만들어진 조직이기 때문에 사전 질의서에 답변한 과제를 수행한 적은 없음. 팀 자체가 내부에 있는 레거시 시스템을 재구축하는 업무를 담당하고 있음. AI 기반으로 활용할 수 있는 것을 개발한다고 이해하면 될 것 같음. 사전 질의서에 적혀 있는 과제는 DDI 전체에서 진행하고 있는 프로젝트이기 때문에 AI 사업2팀에서 진행한 과제가 있을 것임.
	(커니) 전자BG에서 진행하는 5개 과제도 진행하는 중인지?
	(이은광 팀장) 진행하는 팀이 다 다를 것임. 장표에 적혀 있는 대로 프로젝트가 흘러가지는 않아서 전체 콘셉트만 맞춰져 있고 하위 프로젝트는 거의 바뀌었음. 예를 들어서 제품 역설계 및 신뢰성 평가는 진행 중이고 SOI 별 매출액/수량 예측은 안 하고 있음.
	(커니) 해당 과제들이 진행되지 않는 이유가 무엇인지?
	(이은광 팀장) 담당자가 아니기 때문에 모르겠음.
	(커니) 이전 인터뷰에서 듣기로는 Resilient AI S&OP 과제의 경우에는 데이터 준비가 안 돼 있어서 SCM PI부터 하는 걸로 바뀌었다는데 맞는지?
	(이은광 팀장) 데이터 준비보다는 과제의 전체 방향 수립이 안 되어 있는 것임. S&OP 체계라는 것은 DP, MP, FP 같은 흐름이 있는데 그런 흐름을 타기 위해서는 각각 어떤 파트에서 누가 어디까지 해야 하는지 정리가 되어 있어야 하는데 전자BG의 경영 환경이 급격하게 변경되었기 때문에 기존의 시스템이 따라가지 못하는 상황임.
	(커니) 이물 AI Model 운영 환경 구축 및 개발은 어떤 프로젝트였는지?
	(이은광 팀장) 이물 검사는 사진을 찍어서 진행되는데 이 이미지를 바탕으로 이물의 오염도를 판단하는 모델과 이 모델을 분석한 수치가 PDF 형태로 산출되는데 이 PDF에서 데이터를 추출하고 모델 자체를 돌리는 것임. OCR을 사용하는 것이 아니라 PDF에서 파싱하는 것임. 
	(커니) 파싱할 때 사용하는 정해진 툴이 있는지?
	(이은광 팀장) 해당 과제를 진행하기 전에는 없었고 이제 만들었음. 파싱을 진행하면 일부 위치가 변동될 수 있으니, LLM으로 한 번 더 판단하는 로직이 있는 걸로 앎. 직접 구현하지는 않았기 때문에 상세한 내용은 모르겠음. 과제는 해당 데이터 즉 PDF를 파싱해서 DB화하고 관리할 수 있는 게 목적임.
	(커니) DDI에서 AI/Data 과제를 수행하면서 어려웠던 점은 없는지?
	(이은광 팀장) 두산전자BG AI PJT (Phase 1)에서 가장 어려웠던 것은 이메일이었고 그다음으로는 실험실에서 나오는 데이터임. 어떤 회사의 장비를 가지고 추출한 데이터냐에 따라 양식이 다르고, 숫자 레벨과 숫자를 표현하는 정의가 있는 반정형 형식인데 화학식을 JSON으로 처리해 주는 전문 솔루션을 구매해서 데이터를 처리했음. 
	(커니) 이메일을 과제에 활용하기 위해 어떤 프로세스를 거쳤는지?
	(이은광 팀장) 이메일을 전처리한다는 것은 이메일을 스레드 단위로 나누고 스토리 라인을 구성할 수 있도록 묶는 것임. 원하는 데이터가 이메일 안에 있을지 없을지 모르기 때문에 추출하지는 않음. 그래서 구성된 스토리 라인을 바탕으로 이메일 관련된 질문을 하거나 프로젝트 현황을 조회할 수 있도록 에이전트를 활용하는 것임. RAG를 활용해서 잘 검색하게 하고 스토리를 구성하는 기법을 이용하는 방법을 구성하는 것이지 유의미한 데이터를 추출하는 것은 사실상 어려움.
