## [슬라이드 2] 계열사 맥락

### 두산지주 CHRO는 어떤 조직인가

- 두산 그룹 전체의 인사(HR) 기획·운영을 총괄하는 지주사 HR 본부
- 지주사 특성상 각 계열사 HR 데이터를 컨설팅·지원하는 역할 수행 → 계열사 데이터 접근 필요, 그러나 계열사별·기능별(HRM/HRD) 권한 격리 요구가 공존
- 주요 데이터: SAP 기반 정형 인사 데이터(성명·주민번호·평가·발령 이력 등) + 비정형(조직도 PPT, 취업규칙 Word, 교육 이메일)

### AI/데이터 업무 추진 구조

| 역할 | 담당 |
|------|------|
| PM (과제 전체 기획·추진) | CHRO (두산지주 HR) |
| 비식별화·마스킹 기술 지원 | PwC (컨설팅사) |
| 원천 데이터 추출 지원 | DDI (정보시스템팀) |

- PoC 과제 총 **12개** (Raw 코드 8개 + Vibe 코드 4개) 동시 진행, 서브 프로젝트 3개 추가 예정
- 참여 현업 인원: 총 **60명** (전원 비개발자), 과제별 6~7명
- 전체 일정: 5주 (데이터 식별·준비에 2주 배정 / 인터뷰 시점 기준 2~3주차 진행 중 / PoC 완료 목표: 7~8월)

## [슬라이드 3] 사례 1. 인사 민감정보 비식별화·가명처리

**[과제 유형] PoC 전 과제 공통 전처리 업무**

■ **헤드 메시지**: 비식별화 기준이 없어 12개 PoC 과제마다 60명 비개발자가 수작업으로 개별 정의—표준화된 체계 없이 "한 땀 한 땀" 반복.

---

| ① AS-IS 업무 | ② AI 과제로 하려던 것 |
|---|---|
| 민감 정보(이름·주민번호·생년월일 등)가 포함된 인사 데이터를 CHRO PC에서 컨설턴트·SA팀과 협의한 로직으로 수작업 비식별화 후 서버 업로드 | 12개 PoC 과제 전체에 실제 인사 데이터를 활용하기 위해 가명처리 완료 데이터를 랜딩 존 서버에 업로드 |

| ③ 데이터를 보려니 막힌 지점 | ④ 부재한 데이터 주제 |
|---|---|
| 비개발자 현업이 비식별화 방법 자체를 모름 | **F-4 AI 데이터 보안**: 비식별화·가명처리 수준·방법 기준 미정의. 비정형 민감 데이터(이메일) 처리 방법 부재. 복호화 역매핑 체계 없음 |
| 비식별화 수준 기준이 없어 **SA팀에 매번 확인** 필요 | **F-1 데이터 운영관리**: 과제마다 비식별화 파이프라인을 반복 설계하는 표준 체크리스트·프로세스 부재 |
| **과제마다** 가명처리 항목·방법을 개별 정의해야 하는 표준 부재 | |
| 이메일 등 비정형 데이터는 비식별화 방법 자체를 모름 | |
| 랜딩 존 외부로 나간 데이터의 **복호화(역매핑) 방법 없음** | |
| 원본 없이 작업하여 **오류 다발** | |
| 계열사·HRM·HRD 간 열람 권한이 달라 **전체 권한 보유자 없음** | |

> **회의록 인용**
> - (배혜련 팀장) "인사 데이터는 내부적으로 권한이 나뉘어 있어서 민감 정보가 다수 존재함. … 비개발자기 때문에 비식별화하는 방법을 모름." (12행)
> - (배혜련 팀장) "이런 프로젝트를 진행하기 전에 표준화된 프로세스와 체크리스트가 있어서 미리 확인하고 진행할 수 있도록 했으면 좋겠음." (80행)

## [슬라이드 4] 사례 2. 필요 데이터 위치 파악 및 DDI 반복 요청

**[과제 유형] PoC 과제 데이터 수집 단계 (12개 과제 공통)**

■ **헤드 메시지**: 어떤 데이터가 어디 있는지 모르는 상태에서 PoC를 시작—**2주** 예정 데이터 준비가 **2~3주차**에도 완료되지 않고 DDI 반복 요청이 이어지고 있다.

---

| ① AS-IS 업무 | ② AI 과제로 하려던 것 |
|---|---|
| SAP 위주로 정형 데이터 관리. 필요 시 시스템에서 직접 내려받아 사용. 별도 데이터 목록·카탈로그 없음 | PoC 과제별로 필요한 데이터를 SAP·DB에서 식별·수집하여 AI 학습에 활용 |

| ③ 데이터를 보려니 막힌 지점 | ④ 부재한 데이터 주제 |
|---|---|
| 있다고 생각한 데이터가 없거나 못 찾아 **DDI에 반복 요청** | **A-1 데이터 카탈로그**: 어떤 데이터가 어디 있는지 알 수 없어 DDI에 매번 요청. 선제적 데이터 식별 불가 |
| DDI도 데이터를 조합해야 해 **즉시 제공 불가** | **E-1 데이터 Product화**: 동일 데이터를 과제마다 반복 요청·추출하는 구조. 재사용 가능한 데이터 단위(오너 포함) 부재 |
| **1차 수집 이후에도** 필요 데이터가 계속 추가 발생, 범위 반복 확장 | |
| ERD를 몰라(**"ERD가 무엇인지 모름"**) 어느 테이블에 무엇이 있는지 파악 불가 | |
| AI 설계 먼저 그리고 데이터를 역방향으로 찾아 시간 낭비 및 방향 변경 발생 | |

> **회의록 인용**
> - (조욱희 수석) "기본적으로 내려받을 수 있다고 생각했던 데이터가 없다거나 못 찾겠기에 DDI에 요청하면 DDI에서도 데이터가 조합돼서 나와야 하므로 바로 뽑아줄 수 없는 상황이 생김." (20행)
> - (배혜련 팀장) "처음에 생각했던 필요한 데이터의 범위가 계속해서 늘어나는 중임. 이 늘어나는 것들에 대해서도 언제까지 데이터를 확인해야 하는지도 불분명함." (13행)

## [슬라이드 5] 사례 3. 한글 칼럼명 임의 영문 번역

**[과제 유형] PoC 과제 데이터 전처리 단계**

■ **헤드 메시지**: 메타데이터 사전과 용어 표준이 없어, 현업이 한글 칼럼명을 임의 번역—S3 적재 후 실제 데이터와 연결 시 불일치 위험이 잠재된 상태로 진행 중.

---

| ① AS-IS 업무 | ② AI 과제로 하려던 것 |
|---|---|
| SAP에서 내려받은 데이터의 칼럼명은 한글. 테이블·필드 의미를 기술한 메타데이터 사전 없음. 시스템 원본 영문명 미확보 | S3 기반 AI 학습 파이프라인 적재를 위해 칼럼명을 영문으로 변환 |

| ③ 데이터를 보려니 막힌 지점 | ④ 부재한 데이터 주제 |
|---|---|
| 시스템 원본 영문 칼럼명 미확보 → **현업이 한글을 자체 번역** | **A-2 메타데이터**: 테이블·필드 의미·원본 영문명 정보 없어 현업이 임의 번역. 실제 데이터 연결 시 불일치 위험 |
| 임의 번역 영문명과 실제 시스템 데이터 간 **연결 불일치 위험** 잠재 | **A-3 비즈니스 Glossary**: 한글 칼럼명을 어떤 규칙으로 번역할지 용어 표준이 없어 임의 번역 발생 |
| 모듈원이 테이블·필드·값의 의미를 **모르는 상태로 과제 진행** | |
| 영문명↔한글 원본 매핑 문서를 사후 제공—모듈원이 중간중간 숙지해가며 진행하는 비효율 | |

> **회의록 인용**
> - (조욱희 수석) "현재 S3를 사용해서 칼럼명을 영문으로 바꿔야 하는데, 시스템에 원래 있는 영문명을 확보하지 못하고 한글을 자체적으로 영어로 번역해서 사용함. 그래서 나중에 실제 데이터와 연결할 때 문제가 있을 것 같음." (71행)
> - (유지인 수석) "모듈원이 실제로 과제에서 활용하기 위해서는 테이블, 필드, 값이 어떤 의미인지 어려운 상황임." (72행)

## [슬라이드 6] 사례 4. 비정형 데이터 전처리 불가

**[과제 유형] PoC 과제 비정형 데이터 처리 단계**

■ **헤드 메시지**: 조직도 PPT·취업규칙 Word·교육 이메일 등 비정형 데이터를 AI 입력 형태로 전환하는 체계가 전혀 없어, 전처리 작업 전체를 PwC(외부 컨설팅사)에 의존.

---

| ① AS-IS 업무 | ② AI 과제로 하려던 것 |
|---|---|
| 조직도는 PPT 수작업 도식, 취업규칙은 웹 Word 파일, 교육 운영은 메일·PPT·Word로 진행. 비정형 데이터를 AI가 읽을 수 있는 형태로 변환하는 방법·도구 없음 | 비정형 데이터(조직도·취업규칙·교육 이메일)를 PoC 과제에 활용하기 위해 파싱·전처리 |

| ③ 데이터를 보려니 막힌 지점 | ④ 부재한 데이터 주제 |
|---|---|
| PPT·Word 파싱 방법·도구 지식 없어 **PwC에 전적으로 의존** | **B-1 데이터 전처리**: PPT·Word·이메일을 AI가 읽을 수 있는 구조로 변환하는 체계·도구·지식 전무. 컨설팅사 또는 수기 처리에 의존 |
| 외부 도구는 **보안 이유로 차단**—내부 랜딩 존 툴은 기능 제한적 | **F-3 데이터 디지털화**: 조직도·취업규칙·교육 자료가 PPT·Word·이메일 형태로 개인 PC·수작업에 산재. 시스템 연동 가능한 디지털 자산으로 전환되지 않음 |
| 조직도는 인사 시스템 조직 체계와 연동하려면 **수작업만 가능** | |
| 이메일은 개인 정보가 많아 **비식별화 방법 자체를 모름** (전처리+보안 이중 장벽) | |

> **회의록 인용**
> - (조욱희 수석) "PPT 같은 비정형 데이터를 AI가 사용할 수 있게 전환하는 방법이나 도구에 대한 지식이 없어서 컨설팅사에 의존하고 있음. 이 과정에서 외부 tool을 사용하는 것은 보안 측면에서 막혀있기 때문에 내부에서 비정형 데이터를 파싱해야 함." (27행)
> - (조욱희 수석) "이메일처럼 개인 정보 데이터가 많아서 비식별화하는 것이 불가능한 형태도 있음." (40행)

## [슬라이드 7] 사례 5. 조직도 PPT 수작업 관리 및 발령 자동화

**[과제 유형] PoC 과제 (CHRO 15개 과제 중 하나)**

■ **헤드 메시지**: 보고용 PPT 조직도와 SAP 조직 코드 간 표준 매핑이 없어, CHRO 15개 과제 중 발령 자동화 과제가 데이터 준비 단계에서 막혀 있다.

---

| ① AS-IS 업무 | ② AI 과제로 하려던 것 |
|---|---|
| 조직 변경 시 PPT 조직도를 수작업으로 그려 보고 후 SAP에 포지션을 수작업 발령. 시스템 내 도식화된 조직도 없음(hierarchy만 존재). SAP Successor 기능은 가독성 낮아 내려받아 수정 필요 | 조직도(PPT) 업로드 시 자동으로 SAP 발령 등록—또는 시스템 조직 정보의 자동 도식화 |

| ③ 데이터를 보려니 막힌 지점 | ④ 부재한 데이터 주제 |
|---|---|
| PPT 조직명이 SAP 조직 코드와 미연결 (**full name·축약명 혼용, 코드 미부여**) | **B-1 데이터 전처리**: 비정형 PPT 조직도에서 조직 구조 데이터를 추출·변환하는 체계 부재 |
| 시스템 조직 정보와 보고용 PPT 간 **매핑 체계 없어** 수작업만 가능 | **A-3 비즈니스 Glossary**: 조직명 full name·축약명·SAP 코드 간 표준 매핑이 없어 PPT와 시스템 데이터 연결 불가 |
| 비정형 PPT에서 구조화 데이터를 추출하는 방법 없음 | |

> **회의록 인용**
> - (조욱희 수석) "조직도가 조직명이 있고 조직명에 맞는 코드가 지정돼 있을 텐데 CHRO에서 만드는 조직도 체계 보고 자료에는 코드가 지정 안돼 있고 실질적으로 full name은 있는데 이를 축약해서 쓰기도 함. 그래서 데이터 준비 작업이 안 돼 있는 것임." (29행)
> - (배혜련 팀장) "CHRO에서 진행 중인 15개 과제 중의 하나인데 수작업으로 시스템에 발령 내는 것을, 조직도를 넣으면 자동으로 시스템에 등록되는 것을 만들고 싶은 것임." (36행)

## [슬라이드 8] 사례 6. 인사 데이터 접근 권한 분리 및 데이터 카탈로그 보안 딜레마

**[과제 유형] 데이터 거버넌스·보안 운영 이슈 (PoC 추진 과정에서 발현)**

■ **헤드 메시지**: 폴더 권한 관리자 **1명**이 수동 부여하는 파편화된 권한 체계에서, 데이터 카탈로그 구축 자체가 보안 위협이 되는 딜레마—인사 데이터의 존재 목록조차 공개할 수 없다.

---

| ① AS-IS 업무 | ② AI 과제로 하려던 것 |
|---|---|

## [슬라이드 10] 시사점

■ **헤드 메시지**: AI-Ready Data 체계가 갖춰졌다면 데이터 준비 병목을 줄이고 60명 현업이 PoC에 더 집중할 수 있었을 것이다—단, 그 출발점은 보안·비식별화 기준의 선행 정비다.

### 사례별: 체계가 있었다면 달라졌을 것

| 사례 | 갖춰졌을 경우의 변화 |
|------|-------------------|
| 사례 1 (비식별화) | F-4 기준이 있었다면 SA팀 반복 확인 없이 표준 체크리스트로 처리 가능. 비정형 이메일 처리 가이드라인도 존재했을 것 |
| 사례 2 (데이터 위치) | A-1 카탈로그가 있었다면 2주 데이터 준비 기간 내 식별 가능. DDI 반복 요청 구조도 E-1 데이터 Product화로 줄어들었을 것 |
| 사례 3 (칼럼명 번역) | A-2 메타데이터 사전과 A-3 Glossary가 있었다면 임의 번역 없이 원본 영문명 그대로 사용 가능 |
| 사례 4 (비정형 전처리) | B-1 체계가 있었다면 컨설팅사 의존 없이 내부 파싱 가능. F-3 디지털화로 이메일·PPT가 시스템 자산으로 존재했을 것 |
| 사례 5 (조직도 발령) | A-3 표준 매핑이 있었다면 PPT 조직명과 SAP 코드 자동 연결이 가능해 발령 자동화 착수 가능 |
| 사례 6 (권한 분리) | C-2 권한 통제 체계 + F-4 접근제어가 있었다면 카탈로그 구축과 보안이 병행 가능. 1명 수동 관리 구조에서 벗어날 수 있었을 것 |

### 우선 정비 방향

1. **F-4 AI 데이터 보안 (최우선)**: 비식별화·가명처리 수준 기준 정의 + 복호화 역매핑 체계 + 비정형 데이터 처리 가이드라인. 이 체계 없이는 사례 1·4·6이 반복된다.
2. **C-2 데이터 접근 권한 통제 (동시 추진)**: 계열사·HRM·HRD·팀 단위 역할 기반 접근 제어 설계. 폴더 권한 1인 수동 관리 구조의 대체.
3. **A-1 데이터 카탈로그 + A-3 비즈니스 Glossary**: 보안 체계와 병행하여 구축해야 실효성 확보. 카탈로그 공개 범위 자체를 접근 제어 정책으로 설계하는 것이 선결 조건.

> 과장을 피하면 — 체계가 없다고 PoC 자체가 불가능한 것은 아니다. 현재도 PwC·DDI의 지원을 받아 진행 중이다. 다만 12개 PoC 전체가 동일한 준비 병목을 공유하고 있으며, 반복·수작업의 비용이 계속 누적된다는 점에서 정비의 필요성은 분명하다.

> (사례 6) 배혜련 팀장: "지주는 자회사의 HR 컨설팅을 하는 역할을 수행하는 것이기 때문에 다 볼 수 있지만 각 계열사는 계열사의 것만 봐야 하고 HRM과 HRD가 있어서 서로 관련된 것을 보면 안됨." (60행)

> (사례 6) 조욱희 수석: "전체 데이터에 대한 권한을 가진 사람은 없음." (44행)
