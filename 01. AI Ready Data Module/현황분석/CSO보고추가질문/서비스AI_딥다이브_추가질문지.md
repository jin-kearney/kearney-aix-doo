# 두산 밥캣 서비스AI — 사장님 보고용 딥다이브 추가 질문지

> **목적**: 0529 밥캣 인터뷰에서 유영근 팀장이 말씀하신 "서비스AI 구축에 **6개월**이 걸렸고,
> 동일 과제를 지금 하면 **1~2달**이면 된다"는 사례는 *AI-Ready Data 체계가 없을 때의 비용*을
> 가장 선명하게 보여주는 대표 사례다. 사장님 보고 장표의 설득력은 **구체적 숫자와 장면**에서
> 나오므로, 이미 들은 내용을 정확히 이해하고 있다는 전제 위에서 *"몇 개를 · 무슨 오류가 ·
> 어떻게 · 며칠 걸려"* 를 추가로 확보한다.
>
> **대상**: 두산 밥캣 AI CoE팀 유영근 팀장 / 임진열 수석 (서비스AI 직접 수행자)
>
> **사용법**: 각 질문은 "회의록에서 ~~라고 하셨는데, 구체적으로…" 형식으로 구성했다.
> 질문 끝 `★` 는 사장님 장표 헤드라인에 바로 쓸 핵심 수치다. `[코드]` 는 그 수치가 입증하는
> 20개 데이터 주제다. (B-1 전처리 / B-2 해설·주석 / E-3 평가 데이터 / D-3 Prompt 자산화 /
> F-2 생애주기·버전 / C-3 리니지)

## 0. 전제 — 우리가 이해하고 있는 서비스AI (확인용 요약)

회의록 기준으로 우리가 이해한 내용은 아래와 같습니다. **틀린 부분이 있으면 바로잡아 주시고,
맞다면 각 항목 옆의 "구체 수치"를 채워 주시길 부탁드립니다.**

- 서비스AI = 장비 **서비스 매뉴얼(PDF)** 기반으로, "굴삭기 필터 교환 방법" 같은 자연어 질문에
  답을 생성하는 **RAG 챗봇**이다. (line 58)
- PDF 파싱 시 오류가 생겨 **청킹·retrieve 커스텀 솔루션을 자체 제작**해 적용했다. (line 58)
- 매뉴얼은 이미지+텍스트가 함께 전달돼야 정보가 명확한데 멀티모달이 안 돼서, **각 이미지가
  어떤 텍스트(정보)와 연결되는지를 정리·저장**했고, 이때 *"설명이 이미지 위에 있는지 아래에
  있는지"* 혼동 때문에 관리에 시간이 많이 걸렸다. (line 59)
- **전처리 → 답변 생성 → 문제 발견 → 재파싱·retrieve·청킹**을 "하나씩 확인"하며 반복했다. (line 61)
- Liability(법적·안전 책임)와 연관돼 있어 **약 1,000개 질문**으로 하나하나 체크하는 테스트를
  했고, 솔루션 내 **thumbs up/down**으로 품질을 분석했다. (line 72, 74)
- AI 답변에 **"어떤 매뉴얼의 어떤 부분에서 가져왔는지" 소스 출처(Lineage)를 표기**했다. (line 82)
- **밥캣의 첫 AI 프로젝트**라 처음부터 공부해야 했고 그래서 **6개월**, 지금 다시 하면 **1~2달**
  이면 된다. (line 63)

## 블록 1. 대상 매뉴얼 — "무엇을, 몇 개를"

**1-1.** 서비스 매뉴얼을 PDF 기반으로 챗봇을 만들었다고 하셨는데, 대상이 된 **매뉴얼은 총 몇
종(몇 개 파일)** 이었나요? 페이지 합계로는 대략 몇 페이지 규모였나요? ★ `[B-1]`

**1-2.** 매뉴얼은 **장비 모델(예: 굴삭기·로더 등) 몇 종, 몇 개 언어**를 커버했나요? 처음부터
전체를 넣었나요, 아니면 일부 모델로 시작했나요? `[B-1]`

**1-3.** 매뉴얼 한 건당 **이미지(도해·사진·다이어그램)는 평균 몇 개** 정도였나요? 전체로는
연결 정의가 필요했던 이미지가 대략 **몇 개**였나요? ★ `[B-2]`

**1-4.** 매뉴얼 PDF는 **텍스트가 살아있는 디지털 PDF**였나요, 아니면 **스캔본(이미지 PDF)이라
OCR이 필요한 것**이 섞여 있었나요? OCR이 필요한 비율은 대략 몇 %였나요? `[B-1]`

## 블록 2. PDF 파싱·OCR 오류 — "구체적으로 무슨 오류가"

> 회의록: *"PDF 파싱을 하게 되면 오류가 생길 수밖에 없어서 내부적으로 청킹과 retrieve를
> 어떻게 할 것인지 커스텀 솔루션을 만들어서 적용함"* (line 58)

**2-1.** "파싱 오류가 생길 수밖에 없었다"고 하셨는데, **구체적으로 어떤 유형의 오류**였나요?
예를 들어 — 표(table) 구조가 깨지는 문제, 다단(2단·3단) 레이아웃이 한 줄로 섞이는 문제,
머리말/꼬리말이 본문에 끼어드는 문제, 특수문자·단위기호 깨짐, 도면 캡션이 본문과 붙는 문제
중 어떤 것이 **가장 빈번**했나요? ★ `[B-1]`

**2-2.** OCR(스캔본 인식) 단계에서 오류가 있었다면, **어떤 글자·기호에서 오인식**이 많았나요?
(예: 부품 번호의 0/O, 1/l 혼동, 수치·단위, 표 안 숫자) 인식 정확도는 체감상 몇 % 수준이었나요?
`[B-1]`

**2-3.** 기성 파싱 라이브러리(예: 일반 PDF 파서)를 그냥 쓰지 못하고 **커스텀 솔루션을 직접
만들어야 했던 결정적 이유**는 무엇이었나요? 어떤 점이 안 돼서 직접 만들 수밖에 없었나요? `[B-1]`

**2-4.** 청킹(chunking) 기준을 자체 설계하셨다고 했는데, 결국 **어떤 단위로 끊는 것**으로
정착했나요? (예: 절차 단계 단위 / 섹션 단위 / 고정 토큰 길이) 이 기준을 찾기까지 **몇 번의
시도**를 거쳤나요? `[B-1, D-3]`

**2-5.** 파싱·청킹 커스텀 솔루션을 만드는 데만 **순수하게 며칠/몇 주** 정도가 들었나요?
(6개월 중 이 단계가 차지한 비중) ★ `[B-1]`

## 블록 3. 이미지-텍스트 연결 정의 — "위/아래를 어떻게, 얼마나 걸려"

> 회의록: *"각각의 이미지가 어떤 정보와 연결돼야 하는지를 정리해서 저장함 … 이미지에 대한
> 설명이 이미지 위에 있는 것인지 아래에 있는 것인지 혼동이 있기 때문에 이를 관리하는데 시간이
> 많이 걸렸음"* (line 59)

**3-1.** 이미지가 어떤 텍스트와 연결되는지를 정리·저장하셨다고 했는데, 이 작업은 **사람이
수작업**으로 했나요, 아니면 **규칙(스크립트) + 사람 검수** 조합이었나요? 수작업이라면 **투입
인원은 몇 명**이었나요? ★ `[B-2]`

**3-2.** "설명이 이미지 위에 있는지 아래에 있는지"를 정의했다고 하셨는데, **구체적으로 어떻게
판정**했나요? 예를 들어 — 매뉴얼 종류마다 캡션 위치 규칙이 달라서 매뉴얼별로 규칙을 따로
정했나요? 아니면 이미지마다 사람이 보고 위/아래를 직접 라벨링했나요? `[B-2]`

**3-3.** 위/아래 외에도 **좌/우 캡션, 여러 이미지가 한 단계에 묶이는 경우, 한 이미지가 여러
텍스트에 걸리는 경우** 같은 까다로운 패턴이 있었나요? 이런 예외 케이스는 전체의 몇 % 정도였나요?
`[B-2]`

**3-4.** 이미지-텍스트 연결 정의 작업에 **총 몇 주(또는 몇 인-주)** 가 들었나요? "시간이 많이
걸렸다"를 숫자로 표현하면 6개월 중 대략 며칠/몇 주였나요? ★ `[B-2]`

**3-5.** 이렇게 만든 이미지-텍스트 연결 정의가 **다음 매뉴얼·다음 프로젝트에서 재사용 가능한
자산**으로 남았나요, 아니면 매뉴얼이 바뀌면 다시 해야 하는 구조였나요? `[B-2, D-3]`

## 블록 4. 전처리-검증 반복 루프 — "몇 바퀴를 돌았나"

> 회의록: *"전처리를 하면서 답변을 생성해보고 문제가 있으면 다시 파싱, retrieve, 청킹을
> 진행해서 하나씩 확인해 봄"* (line 61)

**4-1.** "문제가 있으면 다시 파싱·retrieve·청킹을 반복했다"고 하셨는데, 만족할 품질에
도달하기까지 **이 루프를 대략 몇 번(몇 회차)** 돌았나요? ★ `[D-3]`

**4-2.** "하나씩 확인"은 **무엇을 기준으로** 문제라고 판단했나요? (예: 답변이 틀림 / 이미지가
안 붙음 / 엉뚱한 매뉴얼 인용) 문제를 찾는 일을 사람이 했나요? `[B-1, E-3]`

**4-3.** 한 번의 루프(전처리 수정 → 재생성 → 확인)에 대략 **며칠**이 걸렸나요? `[D-3]`

**4-4.** 이 반복을 줄여줄 **표준 절차나 체크리스트, 프롬프트 템플릿**이 그때 있었나요? 없었다면,
지금은 그런 자산을 만들어 두셨나요? (이것이 "지금 하면 1~2달" 근거인지) ★ `[D-3]`

## 블록 5. Liability 검증 — "1,000개를 사람이 어떻게 정/오 판단했나"

> 회의록: *"서비스AI 같은 경우는 Liability와 연관이 있는 것이기 때문에 약 1000개 정도의
> 질문을 통해 하나하나 체크하는 테스트를 진행했음 … 현업에서 필요한 질문을 리스트업하고
> 질문지를 업데이트하면서 답변을 추가함 … thumbs up, down으로 체크해서 분석함"* (line 72~74)

**5-1.** "약 1,000개 질문으로 하나하나 체크했다"고 하셨는데, 이 **1,000개 질문지(정답셋)를
만드는 데 며칠/몇 주**가 걸렸나요? **투입 인원은 몇 명**(현업 도메인 전문가 포함)이었나요? ★ `[E-3]`

**5-2.** 한 질문에 대해 AI 답변의 **정/오(맞다/틀리다)를 사람이 어떻게 판단**했나요? 구체적으로 —
미리 만든 **정답지와 한 줄씩 대조**했나요, 아니면 도메인 전문가가 답변을 읽고 **주관적으로
적합/부적합 판정**했나요? 판정 기준(예: 안전 정보 누락은 무조건 오답)이 문서로 있었나요? ★ `[E-3]`

**5-3.** 한 사람이 **질문 1건을 검증(정/오 판단)하는 데 평균 몇 분** 걸렸나요? 1,000건 전체를
한 바퀴 도는 데 **얼마나** 걸렸나요? `[E-3]`

**5-4.** 1,000개 중 **처음 통과율(정답 비율)은 몇 %** 였나요? 틀린 답을 고쳐 다시 검증하는
**재검증 사이클을 몇 번** 돌았나요? ★ `[E-3]`

**5-5.** Liability 영역이라 특히 **"틀리면 안 되는 치명적 질문"**(예: 안전·정비 절차)이 따로
분류돼 있었나요? 그런 항목은 별도의 더 엄격한 기준으로 검증했나요? `[E-3]`

**5-6.** thumbs up/down 피드백은 **파일럿 사용자 몇 명**에게서 받았나요? 수집한 피드백
**건수**와, 그것을 어떻게 개선에 반영했는지 궁금합니다. `[E-3]`

**5-7.** 이 1,000개 평가셋은 **나중에 다른 AI 과제(딜러AI 등)에 재사용**됐나요, 아니면 서비스AI
전용으로만 남았나요? `[E-3, D-3]`

## 블록 6. 소스 출처·버전 관리 — "어느 버전을 인용했는지 알 수 있었나"

> 회의록: *"어떤 매뉴얼에 어떤 부분에서 정보를 가져왔는지 소스 출처가 표기되는데"* (line 82) /
> *"여러 버전의 문서가 어딘가에 존재하지만 어떤 것이 원본인지, 최신인지 등을 단순히 에이전트에
> 연결했을 때는 정확도가 낮음"* (line 91)

**6-1.** 답변에 소스 출처를 표기하셨다고 했는데, 같은 매뉴얼의 **구버전·신버전이 동시에 존재**해
AI가 **옛날 버전을 인용한 사례**가 실제로 있었나요? 있었다면 몇 건 정도 발견됐나요? ★ `[F-2]`

**6-2.** 매뉴얼이 개정되면(부품·절차 변경) **어느 문서가 최신인지 가려내는 작업**을 사람이
했나요? 그 판별에 시간이 얼마나 들었나요? `[F-2, C-3]`

**6-3.** 매뉴얼 **버전·개정 이력을 관리하는 별도 체계**가 그때 있었나요? 없었다면 출처 표기의
신뢰도를 어떻게 담보했나요? `[F-2]`

## 블록 7. 6개월 → 1~2달 — "그 차이를 숫자로"

> 회의록: *"처음 진행했던 것이기 때문에 처음부터 공부했어야 했기 때문에 6개월 정도 걸림.
> 현재 만약 동일한 프로젝트를 한다면 1~2달 정도 걸릴 것 같음"* (line 63)

**7-1.** 6개월을 단계별로 나눈다면 **① PDF 파싱·청킹 / ② 이미지-텍스트 연결 / ③ 전처리-검증
반복 / ④ 1,000개 평가셋 / ⑤ 출처·버전 / ⑥ 순수 AI 로직** 각각에 대략 **몇 %(몇 주)** 씩
들었나요? — *어느 단계가 시간을 가장 많이 먹었는지*가 장표의 핵심입니다. ★ `[전체]`

**7-2.** "지금 하면 1~2달"이라고 하신 근거는 무엇인가요? 구체적으로 — **무엇이 자산으로 남아서**
(전처리 방법·청킹 기준·평가셋·프롬프트·Best Practice 등) 그 시간이 줄어드는 건가요? ★ `[D-3]`

**7-3.** 그 1~2달조차도 **여전히 사람이 다시 해야 하는 작업**(예: 새 매뉴얼 이미지 연결, 신규
평가셋)이 남아 있나요? 만약 전사 차원의 **표준 전처리·주석·평가 가이드**가 처음부터 있었다면
