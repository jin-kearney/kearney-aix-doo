□	두산 밥캣 현업 인터뷰
	(커니) 밥캣에서 진행한 AI/Data 과제가 없었는지?
	(유영근 팀장) 현재 밥캣에서는 다양한 BI 프로젝트, 데이터 분석 프로젝트 등 관련한 AI 프로젝트를 많이 진행하고 있음. 그래서 각각의 Use Case와 프로젝트에 따라서 답변이 달라질 수 있을 것 같아 사전 질의서 답변 자체가 Generic 하게 될 수밖에 없었음. 진행하고 있는 프로젝트의 예를 들자면 AI Trailblazer라는 프로젝트를 런칭해서 100명 정도의 신청자를 리드하고 있음. 프로젝트에 참여하는 인원 중에 일부는 현업에 있는 전문가로 스스로 AI 솔루션을 만들 수 있도록 가이드하고 트레이닝하고 있음.
	(유영근 팀장) 유영근 팀장과 임진열 수석이 소속되어 있는 팀은 AI CoE팀임. AI CoE팀은 크게 3개의 Pillar가 있음. 첫 번째로 AI Development 팀은 주로 AWS 기반으로 주요 비즈니스 Use Case에 대한 커스텀 솔루션을 개발해서 전달하는 역할을 수행함. 두 번째로 AI Execution 팀은 AI Trailblazer 같은 프로그램을 통해서 AI 이니셔티브를 전반적으로 Execution 하는 역할을 수행함. 마지막으로 Business Intelligence 팀은 과거에 대시보드를 만들고 운영하는 팀이었는데 현재는 AI 기반 데이터 분석을 리드하는 팀으로 바뀜. 기존 레거시 시스템을 통해서 비즈니스 분석을 진행했었는데 현재는 AI를 기반으로 역할을 수행함.
	(유영근 팀장) 김민영 수석은 디지털팀 소속으로 전반적인 데이터 플랫폼과 데이터 관리를 담당하고 있음. 
	(커니) 각 팀에서 AI과제를 수행할 때 어떤 업무를 수행하는지?
	(유영근 팀장) Raw 데이터를 현업에서 쓸 수 있게 만드는 데이터 큐레이션은 디지털팀에서 담당하고 있고 큐레이트 된 데이터를 실질적인 비즈니스 솔루션으로 만드는 과정은 AI CoE팀에서 담당하고 있음. 그리고 전반적인 데이터 플랫폼과 거버넌스는 디지털팀에서 담당함.
	(커니) 현재 팀즈에 나와있는 팀명과 방금 전달한 팀명이 다른데 어떤 것인지?
	(유영근 팀장) AI CoE팀이 과거에 데이터 사이언스 팀으로 불렸고, 현재는 변경됨. 팀즈에는 아직 반영이 안 된 것. 기존에는 데이터 사이언스 팀 산하에 BI 팀, Apply Data Science 팀, Core Data Science 팀이 있었는데 업무 Focus가 AI 중심으로 바뀌어 나가면서 변경된 것.
	(커니) 데이터 전체 구조 파악에 시간이 소요됐던 과제가 있는지?
	(유영근 팀장) 항상 소요됨. 밥캣은 오라클 중심의 데이터베이스를 지난 10년간 지속적으로 구축, 운영, 업데이트하고 있음. 예를 들어서 판매 데이터, 오더 데이터, 인벤토리 데이터 같은 기본적인 비즈니스에 필요한 데이터를 체계적으로 정리하여 관리하고 있음. 대시보드 역시 수천 개가 만들어져 운영되고 있어서 Data Maturity 레벨은 높은 편임.
	(유영근 팀장) 최근에 SAP로 전환이 되면서 기존의 데이터를 새로운 ERP 시스템에 맞춰서 리빌딩 하는 과정을 진행 중임. 리빌딩 되는 메인 플랫폼은 Snowflake이고 여기에 ERP데이터와 주요 데이터를 모으고 있음. 그래서 현재는 이 데이터를 모으고 기본적인 데이터 큐레이션이 이뤄지는 단계임. 그래서 Transition이 완료된 데이터는 이해와 접근이 쉬운 편이고 아직 Transition이 되지 않은 데이터는 수작업으로 데이터를 가져와서 가공해야 함. 이는 일시적으로 ERP 플랫폼이 바뀌면서 일어나는 현상이고 전반적으로 데이터 관리 측면의 Maturity 레벨은 높음.
	(커니) 현재 테이블 정의서 같은 것들이 명확히 준비되어 있는 수준인지?
	(유영근 팀장) 내부적으로는 그런 프로세스가 정의되어 있음. 예를 들어서 장비를 판매하는 딜러십의 경우에 딜러십이 담당하고 있는 산업의 사이즈가 얼마인지 질문하면 그 딜러십이 담당하고 있는 지역, 지역별 카운트, 카운트별 산업 정보 등이 다 매핑되기 때문에 개인이 별도 테이블에 있는 데이터를 가져와서 가공할 필요가 없음. 이를 기반으로 마켓 셰어를 계산하는 로직 같은 것도 정해져 있기 때문에 사용자 입장에서는 이미 큐레이트 된 데이터를 사용하면 됨. 그래서 이런 것들을 연결하기 위한 조직도, 타임라인에 대한 구조, KPI 구조 등이 정의 및 운영되고 있음.
	(커니) 해당 데이터 체계는 BI팀에서 다 알고 있는 정보인지?
	(유영근 팀장) 기본적으로 디지털팀에서 Raw 데이터를 가져와서 비즈니스 로직과 Definition에 맞게 Foundation을 구축하는 작업을 수행하고 있음
	(김민영 수석) 디지털 팀 내에도 여러 개의 팀이 있음. 데이터를 관장하는 팀도 있고 플랫폼을 운영하는 팀도 있음. 김민영 수석이 속해 있는 팀은 디지털 전체의 전략 업무를 수행하고 있음. 
	(커니) AI CoE팀에도 BI 팀이 따로 있는데 디지털팀도 따로 하는 팀이 있는 것인지?
	(유영근 팀장) 디지털팀에서는 어떤 정보를 가져오고 가공을 해서 데이터를 쌓는 업무를 진행하고 있고 그런 데이터를 가져와서 리포트 혹은 대시보드로 만드는 플로우는 BI 팀에서만 전담하진 않음. BI 팀에서 주도해서 직접 만들기도 하지만 상당 부분은 각 Function에서 스스로 만들기도 함.
	(커니) 데이터 정의 차이가 발생한 사례가 있는지?
	(유영근 팀장) 데이터 큐레이션 단계에서 정의가 됨. 예를 들어서 포스트 단계에서는 상위 개념의 비즈니스 매트릭을 구성하기 위해 계산이 이루어져야 하는데 그런 계산을 할 때 어떤 필드를 어떤 로직으로 해야 하는지 확인되기 때문에 혼란이 발생하지 않는 것 같음.
	(임진열 수석) 실제 데이터에 Definition 정의가 되어있기 때문에 활용하는 데 문제는 없음. 다만 처음 본 사람의 입장에서는 혼동되는 부분이 있을 수도 있으나 누구에게 데이터를 확인하면 되는지 가이드만 돼 있으면 실제 AI 프로젝트 진행에 큰 문제는 없음.
	(유영근 팀장) 기존의 Practice를 모르는 상태에서 데이터만 봤을 때 누구나 명확하게 답을 찾아낼 수 있는 단계는 아님. 다만 밥캣은 데이터를 지속적으로 운영해 왔기 때문에 담당자를 통해 Definition 등을 찾는 데 어려움이 없음. 또한 SAP 기반으로 새롭게 구축되는 과정에서 다시 한번 Definition 문서화와 메타데이터 입력이 이뤄지고 있음. 
	(커니) Definition 문서화와 메타데이터 입력 같은 작업은 어떤 팀에서 진행 중인지?
	(유영근 팀장) 디지털팀은 이를 테크니컬하게 시스템으로 구현하고 있고 컨텐츠 자체는 현업에서 진행 중이기 때문에 현업에 데이터 오너를 지정하고 있음. 실제 구축하고 파이프라인을 만드는 것은 관련 전문가가 담당하지만, 데이터 오너십은 현업이 갖고 있음. 그래서 주요 데이터 소스별로 데이터 오너가 지정되어 있고 그들이 Definition 정의와 문서화를 담당함.
	(커니) 지정된 데이터 오너는 어느 팀에서 관리하는 것인지?
	(유영근 팀장) 데이터를 가장 많이 쓰는 해당 팀의 담당자가 데이터 오너로 지정이 됨
	(김민영 수석) ERP 프로젝트팀에서 데이터 오너를 지정하고 관리할 것 같음. 현재 계획은 ERP 전환이 끝나더라도 데이터 오너를 유지해서 관리하려고 함.
	(유영근 팀장) 데이터에 대한 Foundation을 만드는 작업이 AI 프로젝트와 일반적인 비즈니스 운영과 크게 구분되지 않는 것 같아서 별도로 만들어야 하는 니즈가 강하진 않음.
	(커니) 품질 보완이 필요했던 사례가 있는지?
	(유영근 팀장) 품질 관련해서는 기존에 보이지 않았던 문제들이 프로젝트 진행 중에 하나하나 확인해 가는 과정에서 발견 및 해결하고 있음. 현재는 새롭게 프로젝트를 진행하고 있기 때문에 안정화되기 전까지 그런 과정이 있을 수밖에 없음. 
	(커니) 고도화 과정에서 휴먼 에러가 발생할 수 있는데 이를 방지하기 위한 프로세스나 거버넌스가 있는지?
	(유영근 팀장) 없음. 에러가 발생하지 않게 하기 위한 별도의 프로세스가 있는 것이 아니라 사용하면서 에러가 발생했을 때 누구에게 컨택하고, 어떻게 이슈를 해결해야 하는지에 대한 프로세스는 있음. 새로운 시스템의 빌드업 단계라서 그렇고, 고도화되면 필요한 단계일 것 같음.
	(김민영 수석) SAP를 구축하면서 데이터에 대한 정의를 다시 하고 있기 때문에 변경에 대한 이슈가 있을 때 변경 프로세스를 통해서 진행할 수 있도록 준비하고 있음. SAP 구축이 끝나고 나서도 데이터 변경에 대한 프로세스를 지속적으로 관리하려고 하고 있음.
	(커니) 해당 프로세스를 시스템화하는 것인지 아니면 방법론으로 접근하는 것인지?
	(김민영 수석) 방법론으로 관리하려고 함.
	(커니) Snowflake에 통합된 데이터와 레거시 시스템에 남아있는 데이터가 어떻게 다른 건지?  
	(유영근 팀장) 레거시 시스템에 남아 있는 것들은 굳이 통합하지 않아도 되는 데이터이거나 우선순위에서 밀리는 것들임. Snowflake에 모든 데이터를 모으는 개념이 아닌 메인 플랫폼으로 중요한 데이터를 모으는 작업임. 비즈니스 Function 별로 우선순위를 정해서 진행 중임.
	(커니) 밥캣의 중요한 데이터라고 하면 어떤 것인지?
	(김민영 수석) 가장 중요한 데이터는 재무 데이터고 재무 데이터의 Snowflake 통합은 완료됨.
	(유영근 팀장) 재무 데이터 같은 비즈니스 Operation 데이터가 가장 중요하고 밥캣은 제조업이기 때문에 오더, 자재, 생산과 같은 데이터가 가장 중요함. 모든 설비 데이터가 올라가는 것은 아님.
	(임진열 수석) 생산 데이터는 관련 팀이 Snowflake에 통합하는 계획을 수립했고 실제 SAP 데이터의 대부분을 Snowflake에 이전하여 관리할 것이라고 전달받음.
	(유영근 팀장) 엔지니어링에서 쓰고 있는 도면 데이터나 R&D 데이터는 올라가 있지 않아서 AI 프로젝트를 진행하면서 시스템에서 가져오는 작업이 있을 수 있음.
	(커니) 비즈니스 Operation 데이터를 먼저 우선순위로 올린다고 했는데 그것을 선정하는 기준이 있는지?
	(유영근 팀장) 비즈니스 Operation을 위한 주요 의사결정에 관련된 데이터를 우선순위로 결정함. 예를 들어서 어떤 모델을 얼마나 많이 생산해서, 언제 줘야 하는지에 대한 의사결정을 하기 위해서는 인벤토리 자재, 오더 등에 대한 데이터가 필요함.
	(커니) 데이터 카탈로그 같은 것이 구축되어 있는지?
	(유영근 팀장) Snowflake에 데이터를 저장하고 있는데 Semantic Layer를 하나씩 만들어가고 있음. 아직까지는 원하는 데이터를 찾을 수 있는 단계가 아님. 데이터를 관리하고 분석하는 담당자가 필요에 의해 원하는 걸 찾을 수 있는 수준임.
	(커니) 이미지나 문서들의 품질 판단 기준이 부족했던 사례가 있는지?
	(유영근 팀장) 서비스AI라고 서비스 매뉴얼을 기반으로 질문했을 때 답을 생성해주는 챗봇 서비스를 만드는 프로젝트를 했음. 예를 들어 굴삭기의 필터를 교환하려면 어떻게 하는지 질문하면 답을 생성하는 것임. 매뉴얼은 PDF로 존재하는데 AI챗봇이 원하는 답을 PDF에서 찾아서 답변할 수 있는 솔루션을 만들고자 함. 이 과정에서 PDF 파싱을 하게 되면 오류가 생길 수밖에 없어서 내부적으로 청킹과 Retrieve를 어떻게 할 것인지 커스텀 솔루션을 만들어서 적용함.
	(유영근 팀장) 매뉴얼은 이미지와 텍스트가 같이 전달돼야 명확한 정보가 전달될 수 있는데 멀티모달로 이를 읽고 답변할 수 있는 상태는 아니기 때문에 PDF에 있는 각각의 이미지가 어떤 정보와 연결돼야 하는지를 정리해서 저장함. 이를 통해 AI 답변에 이미지가 포함될 수 있게 만들었음. 이 과정에서 이미지에 대한 설명이 이미지 위에 있는 것인지 아래에 있는 것인지 혼동이 있기 때문에 이를 관리하는 데 시간이 오래 걸렸음.
	(커니) 그런 관리 작업은 어떻게 진행했는지?
	(유영근 팀장) 전처리하면서 답변을 생성해 보고 문제가 있으면 다시 파싱, Retrieve, 청킹을 진행해서 하나씩 확인해 봄.  
	(커니) 해당 프로젝트를 진행하는 기간이 얼마나 됐는지?
	(유영근 팀장) 해당 프로젝트는 AI 프로젝트 중에서 처음 진행했던 것으로, 처음부터 공부해야 했기 때문에 6개월 정도 걸림. 현재 만약 동일한 프로젝트를 한다면 1~2달 정도 걸릴 것 같음.
	(커니) Legacy 시스템에 저장 돼있는 도면 데이터를 활용한 AI 프로젝트를 진행한 사례가 있는지?
	(유영근 팀장) 진행한 사례가 있긴 있는데 AI가 도면의 내용을 대략 추출하는 수준이지 아직 도면의 내용 자체를 완벽하게 이해할 수 있는 수준은 아님. 그래서 AI가 도면을 읽고 작업을 수행할 수 있는 것을 구현하고 싶었는데 그렇게까지는 못함. 일부 도면 데이터는 도면을 생성하기 위한 소스 데이터가 구조적으로 잘 만들어진 사례가 있음. 예를 들어서 전장 도면 같은 경우에 PDF로 된 전장 도면을 읽고 데이터를 만들어내는 것이 아니라 도면을 만들기 위한 소스 데이터를 기반으로 AI에 넣으면 정확하고 잘 정리된 데이터가 됨. 그래서 소스 데이터를 활용하는 프로젝트가 있음.
	(유영근 팀장) 실제 매뉴얼에 없는, R&D가 보관하고 있는 도면이 실제로 읽고 해석하기 힘듦. 그래서 이상이 있는 부분을 도면의 소스 데이터에서 필요한 부분만 발췌해서 사람이 쉽게 이해할 수 있는 버전으로 재생성해서 보여주는 Circuit Agent를 개발해서 PoC를 진행하고 있음.
	(커니) 형식 정리나 기본 전처리가 필요했던 데이터는 앞서 언급한 사례로 이해하면 될지? 혹은 다른 사례가 있는지?
	(유영근 팀장) 모델을 학습하는 것은 안 했음. 과거에는 Fine Tuning이라고 해서 RAG를 쓰지 않아도 답변을 하는 방법론이 존재했는데 의미 있게 적용되지 않을 것 같다고 내부적으로 판단함. 다양한 모델을 사용해 보고 그중에 좋은 모델을 선택하는 방향으로 진행하고 있음.
	(커니) 데이터 준비가 간단했던 이유가 데이터 품질이 높아서인지? 혹은 과제 범위가 제한적이기 때문인지?
	(유영근 팀장) Unstructured 데이터는 전처리 작업이 필요하지만, Use Case는 주로 보유하고 있는 데이터를 기반으로 진행하기 때문에 없는 데이터를 만들어내지는 않음. 보유하고 있는 데이터를 최대한 활용하는 솔루션을 진행하고 있음. 그래서 대규모 가공이 필요한 사례는 AI 프로젝트를 진행하면서 많이 없었음.
	(커니) AI과제 진행 중 PoC를 했을 때 사용자 피드백을 어떻게 받았는지?
	(유영근 팀장) 사용자가 사용해 보면서 이슈가 있는지 혹은 답변의 퀄리티가 기대 수준에 합당한지를 알아야 개발할 수 있는 것들이고 서비스 AI 같은 경우는 Liability와 연관이 있는 것이기 때문에 약 1,000개 정도의 질문을 통해 하나하나 체크하는 테스트를 진행했음. 
	(커니) AI CoE팀에서 모든 서비스를 알 수 없었을 것 같은데 어떻게 1,000개의 질문과 정답지를 만들었는지?
	(유영근 팀장) 현업에서 필요한 질문을 리스트업하고 질문지를 업데이트하면서 답변을 추가함. 솔루션 자체에서도 질문했을 때 답변이 좋은지 나쁜지를 Thumbs Up/Down으로 체크해서 분석함.
	(커니) AWS QA나 모니터링 프레임워크에 연동해서 진행한 것인지?
	(유영근 팀장) AWS플랫폼으로 가기 전에 진행했음.
	(커니) 서비스AI는 실제 Production 단계인지?
	(유영근 팀장) 이제 Production으로 가고 있는 단계임. 파일럿 단계에서 사용자의 피드백을 받고 필요에 따라서 업데이트하는 수준임. AI 솔루션 개발은 개발하는 과정에서 새로운 피처와 성능이 업데이트되는 주기가 빨라서 특정 스냅샷에서 퍼포먼스를 수집하는 것이 의미가 없음. 개발 단계에서는 수시로 사용자의 피드백을 수집하고 있음.
	(커니) 서비스AI말고 현업 피드백을 수집하고 진도를 많이 나간 과제가 있는지?
	(유영근 팀장) 딜러 AI라고 해서 딜러 퍼포먼스 관련 질문을 하면 AI가 데이터를 분석해서 알려주는 솔루션을 개발했음. 예를 들어서 굴삭기를 대리점에 판매하게 되면 대리점의 실적과 퍼포먼스가 Snowflake에 저장되어 있고 이를 분석하고 모니터링할 수 있음.
	(커니) Snowflake에 올라가 있는 데이터의 출처, 버전, Lineage 관리를 하는지?
	(유영근 팀장) AI 솔루션별로 다른 것 같음. 서비스 AI 같은 경우에는 어떤 매뉴얼에 어떤 부분에서 정보를 가져왔는지 소스 출처가 표기되는데 딜러 AI 같은 경우에는 데이터 소스 자체가 데이터 테이블이기 때문에 소스를 굳이 표시하지 않음.
	(커니) 거버넌스 측면에서 현업과의 협업이 잘 되는 요인이 무엇인지?
	(유영근 팀장) 비즈니스 Stakeholder가 가지고 있는 비즈니스 도메인 지식이 중요한 것 같음. AI 솔루션에서 개발자와 사용자가 분리되는 것이 아니라 데이터에 대한 전문성을 갖고 있는 인원이 현업에 많이 분포되어 있기 때문에 사용자가 개발자가 되는 방식임. 그래서 해당 인원들이 현업과 AI팀의 중간 다리 역할을 잘하는 것 같음.
	(두산지주) 현재 에너빌리티에서는 현업 주도로 AI개발을 진행하려고 하는데 밥캣에서도 그런 움직임이 있는지?
	(유영근 팀장) 있음. 과거에는 AI 개발에 전문성이 요구되었지만, 현재는 플랫폼과 솔루션이 많아서 전문 지식이 없어도 직접 테스트해 보면서 업그레이드할 수 있음. 그래서 개발 영역의 상당 부분이 현업의 Stakeholder가 진행할 수 있고 AI CoE팀이나 디지털팀은 이를 가속할 수 있도록 도우면서 남발되지 않도록 가이드와 Best Practice를 배포하는 데 집중하고 있음. AI Trailblazer도 같은 형식임.
	(커니) 커니에서 진행하는 프로젝트의 산출물에 들어갔으면 하는 내용이 있는지?
	(김민영 수석) 밥캣은 현재 중국, 한국, 미국 등에 퍼져있는데 현업에서 AI에이전트를 만들면서 국가마다 있는 데이터 규정을 어길 수가 있음. 그래서 이를 관리할 수 있는 방법이 있었으면 좋겠음.
	(두산지주) 현재 1차 공유회를 진행한 상태인데 추후에 밥캣에는 어떤 형식으로 공유를 하면 될지?
	(유영근 팀장) 이 내용을 들었으면 하는 Stakeholder가 외국인들이 꽤 많기 때문에 장표나 공유회를 영어로 진행했으면 좋겠음.
	(유영근 팀장) 커니 프로젝트에 부탁하고 싶은 포인트가 있음. 첫 번째로 기존의 Structured 데이터는 자연스럽게 시스템과 플랫폼이 진화하면서 해결될 것 같은데 Unstructured 데이터와 운영에 필요한 문서, 특히 PPT 같은 데이터는 생성하고 AI 솔루션과 연결됐을 때 정확도 이슈가 있을 것 같음. 그래서 AI가 잘 이해할 수 있게 데이터를 생성하는 가이드가 있으면 좋겠음. 두 번째로 이 데이터가 좋은 데이터인지 안 좋은 데이터인지에 대해 구분할 수 있는 방법론이 있으면 좋겠음. 마지막으로 버전 관리가 중요할 것 같음. 여러 버전의 문서가 어딘가에 존재하지만, 어떤 것이 원본인지, 최신인지 등을 단순히 에이전트에 연결했을 때는 정확도가 낮음. 그래서 이런 것들을 직원들에게 잘 교육해서 또 다른 솔루션을 개발하지 않아도 연결만 하면 되는 수준으로 할 수 있으면 좋겠음.
