□	두산 퓨얼셀 현업 인터뷰
	(커니) 사전 질의서에 AI 과제 관련 내용이 없는데 데이터 전처리 관련해서 Pain Point가 있는 것으로 파악됨. 이와 관련한 AI/Data 과제는 없었는지?
	(위호진 수석) 주로 테스트 데이터의 성능적인 아웃풋이 나오면 이를 호기별로 트렌드 분석하는 업무를 하고 있음. 트렌드 관리 중 발생하는 방대한 Raw Data를 처리하기 위해 Chat GPT의 도움을 받으려고 했음. 향후에 데이터 활용도를 높이기 위해 정형화되게 정리하고 있는 단계임.
	(커니) 테스트 데이터가 무엇인지?
	(위호진 수석) 제품의 파워를 올리고 난 후부터 초당 한 줄의 데이터씩 저장되어 30분 단위로 파일 하나가 만들어 짐. 시작부터 테스트 끝까지 2~3일 정도의 데이터가 쌓이고 성능 관련된 부분은 해당 스텝의 데이터를 캡처해 매크로를 돌려서 성능 특성을 뽑아내는 작업을 진행함. 또한 스타트업 같은 테스트 스텝의 Raw Data가 있음. 과거에 진행됐던 테스트 데이터의 매크로 파일을 취합하고 있는 과정임.
	(위호진 수석) 제품이 서비스에 출하돼서 운전하기 위해서는 제품의 성능을 확인해서 나가야 함. 그다음에 중간 단계에 컨디셔닝 하는 프로세스가 있음. 성능 데이터를 호기별로 비교해서 특별히 튀는 게 있는지 등에 대해 시계열 분석을 진행함. 이렇게 트렌드 분석한 시점이 1년 정도밖에 안 되기 때문에 이전 데이터를 좀 더 길게 보려고 과거 데이터를 취합하는 단계임.
	(커니) 그럼 분석한 트렌드 결과를 공장 R&D에 전달하는 것인지?
	(위호진 수석) 스펙은 정해져 있고 스펙 내에서 어떤 트렌드를 보이는지 확인해서 Supplier 업체들의 품질 관리를 피드백해서 결과가 중간으로 오게 하는 과정임. 이는 성능적인 측면이고 Raw Data로는 셧다운이 됐을 때 어떤 특정 태그들의 변화가 있는지 관찰하기 위해서 정리하는 측면도 있음.
	(커니) 어떤 이유로 그 이슈가 발생했는지 원인을 분석하는 것에서 데이터 적인 것들을 분석하는 거라고 보면 되는 건지?
	(위호진 수석) 맞음. 하나의 제품을 테스트하려면 제품의 센서 값이 저장되는 태그가 있는 데 수백 개가 있음. 초당 수백 개의 데이터가 쌓이는데 각각의 태그의 특성을 파악해서 제품 가동 중에 특이 사항이 있는지 등을 향후에 모니터링하려고 계획 중임.
	(커니) 수백 개의 태그가 있다고 했는데 전체 태그 리스트가 관리되고 있는지?
	(이성민 수석) 제품에 PLC라는 부품을 제어하는 태그의 이름은 모델별로 동일하며 서비스에서 쓰는 태그와 공장에서 쓰는 태그는 동일함. 같은 태그 간 매핑에 대한 이슈는 없고 어느 정도 관리는 되고 있음. 사용되는 가스에 따라 시스템의 구조가 다르기 때문에 모델이 바뀌면 태그 이름이 새로 생기거나 할 수는 있음. 그래서 모델별로 정리하고 있음.
	(커니) 전처리 작업 수행하면서 어떤 과정의 소요 시간이 가장 길었는지?
	(이성민 수석) 테스트를 한 사이클 돌리면 2~3일 소요됨. Raw Data의 파일 개수는 40~50개 정도 되고 트렌드를 분석하기 위해서 이를 병합해야 하는데, 이 과정이 오래 걸림. 용량이 너무 크기 때문에 현재 사용하고 있는 엑셀의 매크로 병합 기능으로 처리하는 데 오래 걸림. 또한 설비에서 데이터 로깅할 때 초당 하나에 한 줄씩 로깅되는 파일이 한 번씩 오류가 생겨서 오류를 제거하는 과정이 필요함.
	(커니) 전처리 과정에서 포맷 변환 관련 이슈는 없었는지?
	(위호진 수석) 기계 쪽에서 발생하는 파일은 텍스트나 엑셀 파일이기 때문에 문제없고 성능 데이터를 뽑는 매크로는 호기별 매크로 파일로 되어있음. 특이 사항의 기록은 작업자가 일지에 엑셀 형식으로 기록하고 있고 텍스트 기반의 자유로운 형식이기 때문에 특이 사항을 함께 보기 위해서는 시간이 좀 걸림.
	(커니) 그럼 테스트의 Raw Data를 병합하는 것이 전처리 과정에서 가장 오래 걸렸던 부분인지?
	(위호진 수석) 과거에 매크로 파일을 정리하는 테스트를 했었는데 과거에 있던 파일들은 원드라이브 공유 폴더에 폴더별로 저장이 돼 있고 폴더에 들어가서 해당 매크로 파일을 다운받아 데이터 추출하였음. 일단 폴더에서 로컬 PC로 매크로를 다 빼내는 작업은 했는데 매크로 파일을 일일이 열어서 내부 데이터를 빼낼 수가 없어서 내부 소프트웨어 엔지니어를 통해 3주 정도 소요하여 추출하는 프로그램을 만들어서 추출함. 
	(커니) 그렇다면 이후에 트렌드 분석에는 어느 정도 시간이 걸렸는지?
	(위호진 수석) 트렌드 분석은 제품이 운전되는 특성 등을 알아야 하므로 단순히 태그를 플롯 하기보다 태그 간의 연계성을 봐야 해서 2~3주 정도 걸림. 이후에 정형화된 포맷이 나오고 여기에 데이터를 넣으면 포맷대로 데이터가 플롯 돼서 엔지니어가 판단할 수 있게 됨.
	(커니) 사내에 정형화된 전처리 Tool이 없고 개별적으로 개발해서 사용하고 있는 것인지?
	(위호진 수석) 앞서 언급한 내부 소프트웨어 엔지니어는 데이터 분석 업무에 Involve 된 지 얼마 안 되었고, 이전까지는 파이썬을 활용해서 전처리 프로그램을 만들어 사용함.
	(커니) 전처리가 필요한 파일의 용량은 어느 정도였는지?
	(위호진 수석) 제품 한 대를 2~3일 정도 테스트하면 1기가바이트 정도 나오고 테스트 기간이 길어지면 더 커짐. 
	(커니) 분류 기준 자체는 명확하지만, 폴더별로 나뉘어 있어서 수작업 혹은 코딩으로 모으는 데 오래 걸렸다고 이해하면 될 것 같음. 자동화할 수 있는 규칙이 있는지?
	(위호진 수석) 있음. 첫 번째 행에는 태그 이름이 있고 그 열의 위치에 정해진 날짜에 따라 돼 있어 처리하는 데 큰 어려움은 없음.
	(커니) DAQ시스템이 방금 설명한 원드라이브에 저장되어 있는 매크로 파일들인지?
	(위호진 수석) 아님. DAQ 시스템은 앞서 설명한 초당 한 줄씩 저장되는 데이터임. 공유 폴더에 저장되어 있는 것은 오퍼레이터가 특정 시점의 데이터를 가져와서 매크로를 돌려 성능 데이터를 만든 파일임.
	(커니) DAQ시스템과 테스트 설비마다 있는 로컬 PC가 연동돼 있지 않은 것인지?
	(위호진 수석) 2년 전쯤에 엔지니어가 제품이 셧다운됐을 때 데이터를 로컬로 가져와서 분석했었는데 그렇게 하면 시간이 오래 걸리다 보니 데이터 서버 PC를 마련해서 데이터가 생성되면 서버 PC로 전송해서 DB화했었음. 근데 파일 용량이 크다 보니 서버 환경 문제로 현재는 사용하고 있지 않음.
	(커니) 시스템에 대해 파이프라인을 만들었는데 안 쓰게 된 이유가 서버 용량 문제인지?
	(위호진 수석) 맞음. 그래서 지난주에 내부적으로 NAS 서버 등을 활용해서 증대하는 논의를 진행함.
	(커니) 로컬 PC에서 Raw Data를 다운받아서 성능 데이터를 만드는 데 걸리는 시간은 어느 정도인지?
	(위호진 수석) 현장의 오퍼레이터가 진행하면 10~20분 정도 걸림. 사무실의 엔지니어는 다운 받고 가져와야 해서 30분 정도 걸림.
	(커니) 데이터 이름이 다르다고 사전 질의서에 적었는데 어떤 문제인지?
	(위호진 수석) 모델이 다르면 태그 이름이 달라지는데 Raw Data만 보면 어떤 모델인지 파악할 수 없음. Raw Data의 제목만 봤을 때는 똑같기 때문에 Raw Data를 열어서 태그 이름을 봐야 다른 모델인지 파악할 수 있음.
	(커니) 모델별로 어떤 태그가 있는지 등을 관리하고 있지는 않은지?
	(위호진 수석) 모든 모델의 태그 리스트를 합쳐서 관리하면 좋은데 DAQ시스템에서 처리할 수 있는 태그 수량이 한정돼 있음. 통합하면 수천 개의 태그가 되기 때문에 어려움. 엔지니어 입장에서도 태그가 너무 많으면 엑셀 파일 용량이 커지고 직관적으로 찾기 어려움.
	(커니) 이슈 이력, 작업 특이 사항 등이 비고란에 적혀있다고 했는데 이를 시점별로 통합하거나 디지털화하려는 논의가 있었는지?
	(위호진 수석) 일주일에 한 번씩 데이터 관련된 회의를 내부적으로 진행하는데, 지난주에 관련 얘기가 논의됨. 향후에 해야 한다고 인식하고 있음. 현장에서 일지를 보내주면 그 일지가 이메일의 첨부 파일로 오는데 이를 공용 처리할 수 있는 PC에 계정을 추가해서 그 PC가 메일을 받으면 이메일 내용을 분석하고 이를 엑셀로 정리해 주는 것을 개발하려고 생각하고 있음. 
	(커니) 매주 진행하는 데이터 회의의 참석자는 누구인지?
	(위호진 수석) 품질 쪽 인원임. 트렌드 관리를 하려면 엔지니어가 데이터 취합하는 시간도 오래 걸리기 때문에 이를 자동화할 수 없을지에 대한 논의를 중심으로 진행함. 아이디어 관점에서 얘기했던 부분이고 아직 실행 단계는 아님.
	(커니) 전사 공용으로 사용할 수 있는 AI/Data 과제인 것 같은데 전사적으로는 운영되지 않는 것인지?
	(위호진 수석) 퓨얼셀 전사적으로 하고 있지는 않고 품질팀이 데이터를 많이 다루기 때문에 논의하고 있음. 담당자들이 자발적으로 참여해서 본인의 작업 방식과 해결 방안 등을 교류하면서 최적화하기 위함임. 
	(커니) 데이터를 찾는 과정에서 DB화되지 않아서 현장에서 백업을 받아야 한다고 했는데 백업의 빈도 혹은 항목을 알 수 있을지?
	(위호진 수석) 백업은 Raw Data를 말한 것이고 Raw Data는 정상적으로 과정이 끝나면 볼 필요가 없는데 이슈가 발생했을 때 필요함. 트러블이 생기는 빈도는 10~20% 정도라서 그 비율로 데이터를 받고 분석하고 있음. 직접 현장의 로컬 PC에서 데이터를 받던가 오퍼레이터에게 파일을 뽑아달라고 하는 방식으로 진행함. 
	(커니) MES에서 어떤 변경 작업이 어려웠던 것인지?
	(위호진 수석) 예를 들어서 MES는 매크로 성능 파일을 업로드하면 해당 셀에 있는 데이터를 빼서 DB화 되게 돼 있는데 원하는 데이터 추가가 될 수도 있고 위치가 바뀔 수도 있음. 그래서 이를 MES에 요청해서 바꿔야 함. 그러므로 MES에서 처리되는 데이터를 보기보다는 개인이 매크로를 전체 추출해서 보는 게 편함. MES는 중요한 값만 DB화해서 부가적인 운전 데이터는 볼 수가 없기 때문에 수작업으로 보는 게 편함.
	(커니) DAQ 오류가 발생할 때 누락된 데이터는 어떻게 처리했는지? 
	(위호진 수석) 폴더의 용량이 0킬로바이트인 것은 삭제하고 에러가 떠서 엑셀로 나와야 하는 파일이 DAT 파일로 나오면 다시 진행함. 그래서 에러 창이 뜨면 클릭해 주는 프로그램을 사용해서 재실행해 주는 작업을 하고 있음. 
	(커니) 데이터 이상 값 혹은 단위 불일치와 같은 데이터 품질 문제는 없는지?
	(위호진 수석) 없음. 신제품이 나왔을 때 설정을 잘못한 경우는 있음. 운영 상황에서 데이터 자체의 문제는 없음. 
	(커니) 초기적인 설정을 세팅할 때 프로세스가 있는지?
	(위호진 수석) 정기적인 프로세스는 없음. 매크로 파일을 만들 때 중요한 태그 값이 잘 들어왔는지는 확인함. 변경 점이 없으면 크게 결과가 달라지는 경우는 없음. 태그 리스트라는 파일이 있는데 제품, DAQ와 맞지 않으면 에러가 떠서 확인할 수 있음.
	(커니) 특정 폴더 이동 시 자동 DB화되는 체계가 구체적으로 어떤 것인지?
	(위호진 수석) 앞서 설명한 설비에 Raw Data가 생성되면 서버 PC로 옮겨져 DB화되는 것임. 로깅되는 폴더는 계속 로깅되고 있기 때문에 DB에서 완료가 됐는지 인지할 수 없음. 현재는 서버 용량 문제로 사용하지 않기 때문에 DAQ에서 로깅이 완료되면 파일을 공유 폴더에 이동시키고 있음.
	(커니) 매크로 파일이 원드라이브 폴더에 흩어져 있는데 이를 조회할 수 있는 체계가 있는지?
	(위호진 수석) 성능 데이터를 취합하는 매크로 파일은 폴더 이름 자체가 호기 번호(시리얼 번호)로 돼 있어서 제품 시리얼 번호를 기준으로 검색함. Raw Data는 제품의 호기 번호, 스테이트 번호, 킬로와트 정보 등으로 저장돼 있어서 검색하면 됨. 단지 앞서 설명한 대로 모델을 알 수 없기 때문에 호기 번호와 모델 정보를 매칭하는 추가 작업이 필요함.
	(커니) 호기 번호와 모델 정보를 매칭하는 공용 매핑 테이블이 있는 것은 아닌지?
	(위호진 수석) 예를 들어서 천연가스 모델이면 천연가스 모델 폴더에 호기별로 Raw Data를 모으는 방식임.
	(커니) 이상 데이터가 동일 폴더에 혼재되어서 전처리 시간이 증가했다고 언급했는데 어떤 내용인지?
	(위호진 수석) 현재 업무를 맡기 전에 이전 담당자가 AI 과제를 한 것을 옆에서 지켜본 것임. 이상 데이터를 넣었을 때 자동으로 AI가 탐지하는 것을 하고 싶었는데 학습을 시키려고 하니 폴더 안에는 정상 데이터와 이상 데이터가 혼재되어 있고 어떤 것이 이상 데이터인지 확인할 수 없어서 과거 보고서를 참고해 정리해야 했음. ML을 하려면 샘플 데이터 수량이 많아야 하는데 일일이 분류하기 어려워서 몇 개의 데이터만 가지고 진행함. 향후에는 이벤트가 생기면 그 이벤트의 종류별로 폴더를 미리 만들어서 저장되게 하는 방식 등으로 데이터의 특정 태그로 표시되면 좋을 것 같음.
	(커니) 수기 작업일지가 무엇인지?
	(위호진 수석) 현장에서 작업별로 엑셀 시트에 한 페이지로 만들어 놓는데 그것을 일일이 찾아서 확인해야 함. 양식은 동일하나 제품의 특이 사항이 발생했을 때 셧다운 코드가 50개 정도 있는데 세부적으로 구분되어 있지 않기 때문에 일지를 보고 현상을 파악해서 어떤 셧다운이 발생했는지 추측해야 함.
	(커니) 셧다운 코드의 유형이 왜 구분되어서 작성되지 않는 것인지?
	(위호진 수석) 제품의 소프트웨어에서 상세한 코드가 안 나오고 10개 정도의 대분류로만 나오기 때문에 작업자들이 상세 코드를 모르고 대분류 코드만 적음. 그래서 이를 일지를 보고 다시 트래킹해야 상세 코드를 알 수 있음. 소프트웨어에 Functional Requirement라는 문서에 셧다운 코드가 나와 있는데 이것과 셧다운된 Raw Data 파일을 학습시키면 AI로 세부 셧다운 코드를 파악할 수 있을 것 같음. 장기적으로는 ML을 통해 자동화할 방안을 생각 중임. 그래서 먼저 전처리를 잘해놓자는 생각으로 진행하고 있음.
	(커니) AI작업 시 레이블링에 시간이 많이 걸린다고 했는데 어떤 레이블링을 말하는 것인지?
	(위호진 수석) 셧다운의 상세 코드를 알 수 없어서 수기 작업일지를 확인해서 레이블링하는 것임.
	(커니) 수기 작업일지를 보면 직접 알 수 있는 것인지? 혹은 담당자에게 확인해야 하는 것인지?
	(위호진 수석) 가까운 시점에서 발생한 것들은 카톡 등으로 오퍼레이터와 소통함. 과거의 이력은 일지와 Raw Data를 통해서 비교해 확인함.
	(커니) 레이블링 하는 것은 규칙을 기반으로 하는 것은 아닌 것 같은데 맞는지?
	(위호진 수석) 맞음. 규칙은 따로 없음.
	(커니) 레이블링 상세 코드의 리스트 같은 것은 정해져 있는지?
	(위호진 수석) 코드 자체는 정해져 있음. 개발 쪽에서 미리 정해놓은 규칙대로 셧다운됨. 하지만 셧다운이 발생했을 때 기계에서는 대분류 코드로만 나오기 때문에 직접 Raw Data를 확인해서 어떤 셧다운인지 파악해야 함. 레이블링은 전체 데이터를 다 하지는 않고 필요할 때만 함.
	(커니) 레이블링에 소요되는 시간은 얼마 정도인지?
	(위호진 수석) 품질 과제를 매년 진행하는데 공장 케이스 분석을 하면 과거 이력을 보고 필요한 데이터를 찾는 과정이 하루 정도 걸림.
	(커니) 3월까지 진행했던 AI과제는 뭐였고 결과는 어땠는지?
	(위호진 수석) 다른 공장에서 근무했었기 때문에 상황 정도만 알고 결과는 잘 모름.
	(커니) AI 과제 진행 시 퓨얼셀 내부의 어느 팀과 같이 진행한다는 것이 있는지?
	(위호진 수석) 공식적으로는 없음. 필요한 인원들이 등록해서 참여하는 방식임. 
	(커니) 현재 가장 먼저 AI를 적용해 보고 싶은 공정 혹은 문제가 있는지?
	(위호진 수석) 현장은 24시간 근무를 하고, 엔지니어는 주간 근무만 하고 있음. 그래서 야간에 문제가 생기면 카톡 등이 오는데 챗봇이 이를 대신 답해주는 것이 가능할 것 같음. 셧다운이 생기면 엔지니어가 없어도 오퍼레이터가 처리할 수 있도록 하는 것.
	(커니) AI과제를 한다고 했을 때 준비되어 있는 것이 있는지?
	(위호진 수석) 트러블이 생기면 Trouble Shooting Guide가 있어서 순서도에 따라 분석하는 로직이 있고 Functional Requirement라고 셧다운이 어떤 상황에서 발생하는지 정의한 문서가 있음. 그래서 이 2개의 문서를 학습시켜서 Raw Data를 분석하는 Tool을 만들 수 있을 것 같음.
	(커니) 해당 가이드 문서는 DAQ 같은 시스템의 Raw Data를 처리하는 가이드가 잘 되어있는 것인지?
	(위호진 수석) 문서 자체는 로직적으로 되어있고 Functional Requirement 문서대로 코딩이 돼 있어서 사용 가능함.
	(커니) 가이드 문서의 유지 보수 등은 잘 되고 있는지?
	(위호진 수석) 제품을 개발한 원천 기술을 가지고 있는 미국 회사에서 주기적으로 업데이트해 주고 있는데 큰 변화는 없음.
	(커니) 데이터 버전, 전처리 등은 기록하고 있는지?
	(위호진 수석) 안하고 있음.
	(커니) 거버넌스 측면에서 필요하다고 느끼는 부분에 대해 자세히 설명해 줄 수 있는지?
	(위호진 수석) 어떤 기술이 필요하고 무엇을 진행해야 하는지를 알고 있으면 과제를 좀 더 잘 수행할 수 있을 것 같음. 하지만 과제로 하려면 공식적인 업무가 되어서 엔지니어들이 꺼리는 것 같음. 방향성을 제공해 주면 도움이 될 것 같음.
	(커니) 교육 등을 주도하는 팀은 아예 없는 것인지?
	(위호진 수석) 분야가 방대하다 보니 특정 주제의 강좌를 만드는 것은 좀 어려워 보임.
	(두산지주) 퓨얼셀은 수소 연료 전지 때문에 국가 핵심 기술에 선정되어 영향을 받는 것으로 아는데 작업하면서 해당 기준에 의해 데이터 혹은 시스템 클라우드를 사용하지 못해 AI 활용에 제약이 있었던 적이 있는지?
	(위호진 수석) 품질 데이터를 Chat GPT를 활용해서 분석하고 찾는 것을 생각해 봤는데 품질 데이터를 직접적으로 올려도 되는지 잘 모르겠고 해당 사항을 어디에 물어봐야 하는지 알 수 없음. 현재는 두산 AI Chat만 사용해서 코딩하는 용도로만 사용하고 있음. 
	(커니) 퓨얼셀 내부에서 어느 정도 AI가 필요하다고 느끼는 공통적인 컨센서스가 이루어지고 있는지?
	(위호진 수석) 대부분 필요하다고 인지하고 있으나 AI를 떠올리면 좀 크게 생각해서 업무 외적으로 생각하는 것 같음. 본인이 하기에는 어려운 것이라고 생각하는 것 같음.
	(커니) AI/Data 관련해서 질문, 자문할 수 있는 담당자 혹은 조직은 없는 건지?
	(위호진 수석) 없음.
	(커니) 품질팀 말고 퓨얼셀 전사적으로 공통 인프라를 사용하는 체계는 없는 것인지? 
	(위호진 수석) 품질 데이터는 MES를 통해서 서버에 저장되고 있는데 Raw Data 같은 파일은 로컬 PC에 저장돼 있어서 같이 서버에 올라가면 좋을 것 같으나 아직 그 단계는 아닌 것 같음. 원래 원드라이브에 저장하라고 전달받아서 저장하던 중 용량이 부족해서 용량 증대 요청도 했음. 또 검색 시간이 오래 걸리고 해서 데이터 백업 목적으로 NAS 서버 얘기가 나왔음. 
	(커니) DB관리 등은 조직적으로 어떻게 나누어져 있는지?  
	(위호진 수석) 자세히 모르겠지만 MES 데이터 변경 같은 것은 DDI에 요청해야 함. 퓨얼셀에 IT 관련 담당자는 있는 걸로 아는데 데이터 측면에서는 어떤 일을 하는지 모름. 
□	두산지주 질의응답
	(커니) 퓨얼셀에는 IT관련 조직은 없는지?
	(두산지주) 퓨얼셀은 전통적으로 IT 담당 부서가 아예 없었음. 이전에 관련 업무를 해본 담당자가 있던 걸로 아는데 현재는 퇴사한 상태임. 그래서 이 업무에 대해 아예 공백이 생겼고 나름 관리한다고 다른 팀들이 업무를 배정해서 하는데 총체적으로는 관리가 안 됨. 모든 시스템이 MES를 통해 운영 관리를 하므로 정확한 담당자는 모르지만, 커뮤니케이션하는 방식임.
	(커니) 1차 공유회에 퓨얼셀이 참여하지 않았던 것 같은데 맞는지?
	(두산지주) 참여자 18명에는 포함되지 않았음. 참여자 18명은 AI-ready Data 프로젝트에 참여하겠다는 의사를 밝힌 사람들이고 그 외에는 단지 정보를 공유해 주는 사람들임. 그래서 18명은 사전에 설명을 해서 사인업을 한 사람들임. 로보틱스와 퓨얼셀은 IT 담당 부서가 아예 없어서 공식적으로 커뮤니케이션할 수 있는 채널이 없음. 그래서 공유받는 수준에서 마치는 것임. 로보틱스는 독단적으로 프로젝트를 진행할 것이라고 해서 공유회에 따로 초대는 못 했고 내일 따로 얘기를 나눌 예정임.
